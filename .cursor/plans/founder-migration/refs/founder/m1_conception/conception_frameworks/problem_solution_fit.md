# Problem-Solution Fit Canvas

**Purpose:** Build a Problem-Solution Fit Canvas that proves your proposed solution is a natural, credible response to a well-understood problem in a real customer context.

**Context:** Use in M1 Conception, Step 4. MUST complete Working Backwards (PR/FAQ), JTBD analysis, project memo, and M1 founder log first.

---

## Framework Overview

Problem-Solution Fit Canvas tests whether solution truly fits specific customer problem in concrete context. Forces anchoring on single segment/situation, mapping problem as experienced, documenting current behaviors/alternatives, respecting constraints, and articulating solution concept grounded in realities.

---

## Task Structure

High-level view of framework execution:

| Task | Goal | Key Inputs | Key Outputs |
|------|------|------------|-------------|
| 1. Consolidate Inputs And Define Scope | Turn Working Backwards + JTBD into a precise problem-space brief for a single segment and use case | Working Backwards document, JTBD analysis, project memo draft | Problem-Space Brief, in-scope/out-of-scope definition |
| 2. Map Problem, Triggers, And Emotions | Describe the problem as it is experienced in specific moments, including emotional states | Problem-Space Brief, PR narrative, JTBD "struggling moments" | Canvas blocks: Problem, Triggers, Emotions (before/after) |
| 3. Map Alternatives, Behaviours, And Constraints | Understand what customers do today and what limits adoption of new solutions | JTBD alternatives, market observations, founder intuition | Canvas blocks: Available Solutions, Behaviour, Channels, Constraints |
| 4. Articulate Proposed Solution In Context | Define the solution concept as a response to observed behaviours and constraints | Canvas problem/behaviour/constraint blocks, PR solution description | Canvas block: Our Solution (problem-solution level, not backlog) |
| 5. Extract Critical Assumptions And Risks | Make explicit the beliefs that must hold for real problem-solution fit | Full draft canvas, Internal FAQ from Working Backwards | Canvas block: Critical Assumptions (tagged and prioritised) |
| 6. Finalise Canvas And Connect To M1 Artefacts | Freeze the canvas and wire it into Lean Canvas, 5 Whys, project memo, and founder log | Final canvas, project memo, M1 process doc | Saved canvas doc, updated memo, updated log, inputs to Lean Canvas and 5 Whys |

## Task 1: Consolidate Inputs And Define Scope

**Goal:** Produce a concise, coherent Problem-Space Brief that defines exactly which customer segment and situation this canvas covers.

**Inputs:**

| Input | Source | Required |
|-------|--------|----------|
| Working Backwards document (Press Release + External and Internal FAQs) | M1 Working Backwards framework | Yes |
| JTBD analysis (jobs, progress goals, current alternatives) | M1 JTBD framework | Yes |
| Initial project memo (Problem and Solution drafts, if any) | M1 Step 1 output | No |

**Action:**

1. From the Working Backwards document, extract:
   - The **primary customer archetype** (segment).
   - The **problem description** as stated in the Press Release.
   - Any contextual details and objections from the FAQs.
2. From JTBD analysis, extract:
   - The primary **job statement** in "When / I want to / so I can" form.
   - The list of **current alternatives** and situations where they appear.
3. Decide the **scope** of this canvas:
   - Choose one primary **customer segment** (job title, company size, industry, geography).
   - Choose one **repeating situation or use case** where the problem appears.
4. Write a **Problem-Space Brief** (max half page) that includes:
   - A one-sentence description of the customer segment.
   - A one-paragraph description of the typical situation/context.
   - A problem statement that does not mention your product or solution.
5. Explicitly document:
   - **In scope:** segment and situation this canvas covers.
   - **Out of scope:** other segments or situations you are deliberately ignoring for now and why.

**Output:**

| Output | Format | Purpose |
|--------|--------|---------|
| Problem-Space Brief | Short narrative block at top of canvas | Locks in who and what context this canvas is about, preventing segment creep |
| Scope Notes (in/out) | Bulleted list | Prevents mixing multiple segments/use cases into one canvas |

**Validation:**

How to know you are done correctly:

- [ ] You can name at least **three real companies or individuals** who fit the chosen segment and situation.
- [ ] The problem statement can be read **without referencing your solution** and still makes sense.
- [ ] A neutral reader can restate the segment, situation, and problem after one read of the brief.
- [ ] You have written at least one explicit **out-of-scope** segment or situation.

## Task 2: Map Problem, Triggers, And Emotions

**Goal:** Describe the problem as it is actually experienced by customers, including when it appears and how they feel before and after it is resolved.

**Inputs:**

| Input | Source | Required |
|-------|--------|----------|
| Problem-Space Brief | Task 1 | Yes |
| Working Backwards Press Release and External FAQ | M1 Working Backwards framework | Yes |
| JTBD "struggling moments" / context notes | M1 JTBD framework | Yes |

**Action:**

1. List **3–7 concrete situations** where the problem shows up ("moments in time"). For each, describe:
   - The observable context (when, where, with whom, using which tools).
   - The negative consequence if the problem is not handled.
2. For each situation, identify the **trigger to act**:
   - What specifically makes the customer stop tolerating the status quo and do something?
   - Capture both internal triggers (frustration, fear, pressure) and external triggers (deadlines, audits, incidents).
3. For each situation, capture **emotional states**:
   - Before: how the customer typically feels while the problem is active.
   - After an adequate resolution: how they would like to feel instead.
4. Synthesize these into concise canvas blocks:
   - **Problem:** 3–5 sentences describing the recurring pain in situ.
   - **Triggers:** bullet list of the main triggers to action.
   - **Emotions:** bullet list of typical before/after emotional states.

**Output:**

| Output | Format | Purpose |
|--------|--------|---------|
| Problem block | Canvas section | Encodes the problem as lived experience, not abstract category labels |
| Triggers block | Canvas section | Reveals when and why customers become willing to consider change |
| Emotions block | Canvas section | Ensures solution design respects emotional as well as functional jobs |

**Validation:**

- [ ] Each problem description is anchored in a **specific situation and trigger**, not generic language.
- [ ] Emotional states are plausible for this segment; you can imagine exact sentences a customer would say.
- [ ] You can trace each situation or trigger back to something in your PR, FAQ, or JTBD notes.
- [ ] At least one situation describes a case where customers currently **do nothing** despite pain.

## Task 3: Map Alternatives, Behaviours, And Constraints

**Goal:** Understand what customers currently do to cope with the problem, which solutions they use, and what constraints limit adoption of new solutions.

**Inputs:**

| Input | Source | Required |
|-------|--------|----------|
| JTBD analysis (current alternatives and behaviours) | M1 JTBD framework | Yes |
| Any informal market observations or interviews | Founder experience, research notes | No |
| Problem, Triggers, and Emotions blocks | Task 2 | Yes |

**Action:**

1. List **current behaviours** when the problem appears:
   - What does the customer do first, second, third?
   - Include low-tech workarounds (spreadsheets, emails, manual steps, shadow IT).
2. List **available solutions**:
   - Direct competitors, adjacent tools, processes, or organisational "solutions" (like throwing more people at it).
   - For each, write one line on why it is unsatisfying for this segment and situation.
3. Identify **constraints and limitations** that affect adoption of new solutions:
   - Financial (budget, cost structure).
   - Time and attention (competing priorities, change fatigue).
   - Skills and tools (technical literacy, access to systems).
   - Organisational and political (approvals, compliance, IT/security).
4. Map **channels of behaviour**:
   - Where behaviours and workarounds occur: tools, systems, physical locations, communication channels.
   - Note existing touchpoints that could become channels for your solution.
5. Convert findings into canvas blocks:
   - **Available Solutions** (including "do nothing").
   - **Behaviour** (key patterns and coping strategies).
   - **Channels of Behaviour** (where problem-related actions live).
   - **Customer Constraints / Limitations**.

**Output:**

| Output | Format | Purpose |
|--------|--------|---------|
| Available Solutions block | Canvas section | Prevents reinventing or underestimating existing options |
| Behaviour and Channels blocks | Canvas sections | Ground solution in actual workflows and touchpoints |
| Constraints block | Canvas section | Forces explicit design around what customers cannot or will not do |

**Validation:**

- [ ] For each major behaviour, you can describe how often it occurs and in what context.
- [ ] For each alternative, you can state at least one **concrete drawback** for this segment.
- [ ] Constraints list is specific enough to invalidate or reshape certain solution ideas.
- [ ] You can explain convincingly why the problem has not already been solved by existing options.

## Task 4: Articulate Proposed Solution In Context

**Goal:** Define the solution concept at the right level of abstraction, explicitly as a response to observed behaviours, alternatives, and constraints.

**Inputs:**

| Input | Source | Required |
|-------|--------|----------|
| Problem, Triggers, Emotions, Alternatives, Behaviours, Constraints blocks | Tasks 2–3 | Yes |
| Working Backwards Press Release solution description | M1 Working Backwards framework | Yes |

**Action:**

1. Rewrite the solution description from the Press Release in terms of:
   - Which **specific behaviours** it replaces, augments, or automates.
   - Which **constraints** it respects, relaxes, or shifts (for example, from capex to opex).
2. Describe the solution concept using:
   - The **core mechanism** of value creation and delivery (what fundamentally changes in how the customer gets the job done).
   - The **new customer journey** in the same situations identified in Task 2.
3. Deliberately avoid feature lists and implementation details:
   - Focus on what the customer does differently and what outcome they get, not on screens or architecture.
4. Ensure every major element of the solution can be traced back to:
   - A problem situation or trigger.
   - A behaviour or alternative you intend to replace.
   - A constraint you are designing around.

**Output:**

| Output | Format | Purpose |
|--------|--------|---------|
| Our Solution block | Canvas section | Captures problem-solution concept level, before MVP backlog or detailed design |

**Validation:**

- [ ] For each major element of the solution, you can answer: "This exists because of **which** behaviour or constraint?".
- [ ] If you removed a behaviour or constraint from earlier tasks, some part of this solution would no longer make sense.
- [ ] A neutral reader can understand the solution and how it changes customer behaviour **without seeing a feature list**.

## Task 5: Extract Critical Assumptions And Risks

**Goal:** Turn implicit beliefs embedded in the canvas into explicit, tagged assumptions that will drive later validation work.

**Inputs:**

| Input | Source | Required |
|-------|--------|----------|
| Completed draft of all previous canvas blocks | Tasks 1–4 | Yes |
| Internal FAQ (especially economics and feasibility sections) | M1 Working Backwards framework | Yes |

**Action:**

1. For each major relationship in the canvas, write explicit "This assumes that…" statements, for example:
   - Customers experience the problem with at least a certain frequency or intensity.
   - They are willing and able to change the current behaviour.
   - They have or can get the budget, skills, and approvals needed.
2. Classify each assumption as:
   - **Behavioural** (about customers and their context).
   - **Technical** (about feasibility and performance).
   - **Economic** (about pricing, costs, market size, buying process).
3. Prioritise 5–10 **most critical assumptions** that, if wrong, would destroy problem-solution fit.
4. Cross-reference any assumptions already surfaced in the Internal FAQ:
   - Keep reference IDs or short labels aligned so M2 Validation and Leap of Faith mapping can reuse them.
5. Create a concise **Critical Assumptions** block in the canvas, listing each assumption with:
   - Category, short label, and one sentence description.

**Output:**

| Output | Format | Purpose |
|--------|--------|---------|
| Critical Assumptions block | Canvas section | Becomes explicit feed into M2 Validation and Leap of Faith frameworks |

**Validation:**

- [ ] You have at least one assumption for each major canvas area (problem, behaviours/alternatives, solution, constraints).
- [ ] You can imagine concrete tests or signals that would falsify each assumption.
- [ ] The top 3–5 assumptions feel uncomfortable: if they are wrong, most of your canvas collapses.

## Task 6: Finalise Canvas And Connect To M1 Artefacts

**Goal:** Finalise the Problem-Solution Fit Canvas as an artefact and wire its outputs into Lean Canvas, 5 Whys, the project memo, and the founder log.

**Inputs:**

| Input | Source | Required |
|-------|--------|----------|
| Final Problem-Solution Fit Canvas blocks | Tasks 1–5 | Yes |
| Project memo | M1 Step 1 and Step 7 | Yes |
| M1 Conception process doc | `conception_process.md` | Yes |

**Action:**

1. Create or update the canvas document in the project:
   - Save it under `[project]/docs/founder/conception/problem_solution_fit.md` (or similar agreed naming).
   - Ensure all blocks (Problem-Space Brief, Problem, Triggers, Emotions, Alternatives, Behaviours, Channels, Constraints, Our Solution, Critical Assumptions) are present.
2. From the canvas, derive inputs for **Lean Canvas** (Step 5 of `conception_process.md`):
   - **Problem** and **Customer Segments** directly from Tasks 1–3.
   - **Solution** from Task 4.
   - Partial input into **Unique Value Proposition** and **Channels** from Tasks 2–3.
3. Feed the refined problem into **5 Whys** (Step 6 of `conception_process.md`):
   - Use the problem block from Task 2 as the starting problem statement.
4. Update the **project memo**:
   - Problem section: synthesize from Tasks 1–3.
   - Solution section: synthesize from Task 4.
   - Tenets section: derive guiding principles that encode critical assumptions and constraints.
   - Progress > Conception subsection: list the Problem-Solution Fit Canvas as completed, with date and link.
5. Update the **M1 founder log**:
   - Add at least one entry summarising the main problem-solution thesis.
   - Add entries for the most critical assumptions and any major pivots in problem or solution definition.

**Output:**

| Output | Format | Purpose |
|--------|--------|---------|
| Problem-Solution Fit Canvas doc | Markdown document (applied framework) | Canonical record of problem-solution hypothesis for this segment and situation |
| Updated Lean Canvas, 5 Whys, project memo, founder log | artefacts | Ensures the canvas is operationalised across the rest of M1 |

**Validation:**

- [ ] Lean Canvas can be completed using only information already present in PR/FAQ, JTBD, and this canvas.
- [ ] 5 Whys starts from the problem as framed in the canvas, not a different abstraction.
- [ ] Project memo's Problem and Solution sections are consistent with the canvas.
- [ ] Founder log has at least one entry that clearly references problem-solution fit insights or decisions.

## Pitfalls

**Overloading One Canvas With Multiple Segments**

NEVER cover multiple customer types in one canvas. MUST enforce narrow scope—one primary segment and one recurring situation per canvas.

**Feature-Led Solution Definition**

NEVER jump from problem to feature backlog. MUST force each solution element to trace back to behaviour, alternative, or constraint.

**Ignoring Inconvenient Constraints**

NEVER downplay IT, compliance, budget, or political constraints. MUST surface constraints explicitly and let them reshape solution.

**Treating The Canvas As Static**

NEVER create once and forget. MUST treat as versioned hypothesis and update when assumptions are validated or falsified.

**Substituting The Canvas For Customer Contact**

NEVER fill canvas from intuition only. MUST mark unvalidated parts and design M2/M5 learning activities to confront with real customers.

## Integration

**Prerequisites:** MUST complete Working Backwards PR/FAQ, initial JTBD notes, project memo, and M1 founder log.

**Builds on:**

| Framework | Output Used | How Used |
|-----------|-------------|----------|
| Working Backwards (PR/FAQ) | Customer, problem, solution narratives; Internal FAQ assumptions | Provides initial customer/problem framing and internal economics/feasibility assumptions |
| Jobs-To-Be-Done | Job statements, struggling moments, current alternatives | Supplies real-world context for behaviours, alternatives, and desired progress |

**Feeds into:**

| Framework | Output Provided | How Used |
|-----------|-----------------|----------|
| Lean Canvas | Problem, Customer Segments, Solution, partial UVP and Channels | Populates core blocks with grounded problem-solution hypotheses |
| 5 Whys | Refined problem statement | Starting point for root-cause analysis in Step 6 of M1 |
| Project Memo | Problem and Solution sections, Tenets, Progress > Conception | Ensures narrative memo stays aligned with the latest problem-solution thesis |
| Leap of Faith / Assumption Mapping (M2) | Tagged critical assumptions | Seeds validation backlog and experiment design |

**Workflow Position:** Within M1 Conception, Problem-Solution Fit Canvas sits after Working Backwards and JTBD and before Lean Canvas and 5 Whys.

## Success Criteria

This framework is complete when:

- [ ] A Problem-Solution Fit Canvas document exists for the chosen segment and situation, with all core blocks filled.
- [ ] The solution description in the canvas can be fully explained in terms of previously mapped behaviours, alternatives, and constraints.
- [ ] Lean Canvas and 5 Whys can be completed without redefining the core problem or solution.
- [ ] At least five explicit, testable critical assumptions about problem-solution fit are documented and tagged.
- [ ] The project memo's Problem, Solution, and Tenets sections reflect the canvas, and the M1 founder log contains at least one entry referencing problem-solution fit.

## For AI Agents

**Execution Context:** When executing this framework with a user:

1. MUST confirm that Working Backwards and at least basic JTBD artefact exist. If not, MUST help user create minimal versions before proceeding.
2. MUST read this framework guide and M1 Conception process (`conception_process.md`) before starting.
3. MUST track progress explicitly: "Currently in Task [N]: [Task Name]".
4. MUST push for specificity in situations, behaviours, and constraints. NEVER accept vague descriptions as complete.
5. MUST encourage user to log major changes in problem or solution definition in M1 founder log and update project memo.

**Working With Tasks:**

- MUST treat each task as atomic. NEVER move on until validation checklist is satisfied or consciously deferred with explicit note.
- When critical assumption emerges, MUST tag it clearly and suggest how it might be validated in M2 and M5.
- If you detect multiple segments or use cases being mixed, MUST propose splitting into separate canvases rather than weakening current one.
