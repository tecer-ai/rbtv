# Coding

## Purpose

This module solves **trust in "done"** for coding tasks: agents declare done on self-graded, presence-shaped proof (tests green, element renders) that a non-technical owner cannot audit, so broken features surface under the owner's hand at point of real use. `rbtv-done-gate` makes done purchasable only with user-fidelity evidence — weight-graded, so trivial tasks record proof inline and only substantial work writes a disk sheet. It also runs a Contract-time drivability check: agents that hit a surface they cannot drive (a native OS dialog, an isolated-run config that leaks to real data, a fused output where you can't tell which arm fired) used to re-invent the same workaround at done-time on every project — the gate now moves that discovery to build-time and transmits the known seam patterns so the feature is built drivable from the start.

> **Moved:** plain-language code communication (`rbtv-non-technical-user`) moved to the new **communication** module (2026-06-13) so it is electable independent of coding. See [modules/communication.md](./communication.md).
>
> **Moved:** `rbtv-commit` (guided git commits) moved to the **core** module — committing with hygiene is a foundation concern, not a developer-only one. See [modules/core.md](./core.md).
>
> **Retired:** the `rbtv-coding-discipline` skill that used to live here (don't overcomplicate, don't assume silently, don't touch more than asked, define success criteria) was generalized into the always-on **reasoning** rule's *Execution Discipline* section (core module). Those guardrails apply to all artifact work — vault files, docs, plans — not only code, so a code-scoped skill was the wrong shape.

## Scoping

`rbtv-done-gate` loads always-on once this module is installed — a THIN trigger layer (≈420 words). Its heavy body lives in a protocol (`coding/skills/done-gate/done-gate-protocol.md`, ≈2,200 words) loaded **only when a coding task fires**, via the `rbtv-done-gate` skill loader — so non-coding sessions stop paying the body's words while coding sessions keep full enforcement (the rule's Load Gate makes the load mandatory and the exempt decision stays in the always-on rule). The module is a second scoping boundary: a workspace that does no coding **omits the `coding` module at install** and avoids even the thin always-on load. This is an install-time election the module-granular manifest already supports — not a runtime guess, so no protection is ever silent when a coding task fires. (Plain-language code communication now lives in the separate **communication** module.)

## Components

### `rbtv-done-gate`

- **What**: A passive rule that gates every done-claim on coding tasks. At task intake the agent drafts outcome criteria in plain language ("when done, the owner can {gesture} and {visible result} happens") and confirms them in one pass — the owner never writes formalism. At that same Contract moment it runs a per-criterion **drivability check** before any code is written: for each owner-observable outcome, is its surface something the agent can drive and observe with a real gesture? If not, the agent must build a **test seam** for that surface as part of the feature — a guarded path that supplies or exposes only the unreachable part while the human-facing gesture stays real (three evidenced seam patterns ship: native OS dialogs/widgets, isolated-run config isolation, fused/composed output). One surface is pre-classified at Contract: a live multi-turn interactive workflow-agent / persona session (`/sb-investor`, `/sb-tutor`, mode-routing via a fresh skill invocation) is marked `un-seamable: interactive-agent-session → owner-UAT` and exercised by the owner on a real session — never claimed `drivable`. Before declaring done, the agent exercises each criterion the way the owner will use the feature (real application running whole, owner's actual data when it exists, visible browser + real gestures for UI) and records per-criterion evidence — INLINE in the done message by default, persisting no files; a disk sheet (with capture files) is written ONLY when the owner explicitly asks for one. When a criterion's proof is inherently a file (UI screenshot, measured geometry) and no sheet was requested, the agent captures transiently to take the measurement, reports the measured value inline, and persists nothing — or marks it `unexercisable` for the owner to verify by hand. No evidence → done is undeclarable; a failed row → back to build; unexercisable rows must be surfaced in the done message. `held-surprising` rows — where a criterion holds technically but via a non-obvious mechanism the owner must judge — are also surfaced in the done message; the owner decides acceptance.
- **When to use**: Always active once the coding module is installed. Fires on every coding task except an explicit exempt list (typos, formatting, comments, zero-behavior cleanups, runtime-invisible config). Non-code work is out of scope.
- **How to invoke**: Automatic. No trigger. The thin always-on rule loads passively; the moment a non-exempt coding task starts it fires the **Load Gate**, which invokes the `rbtv-done-gate` skill to load the heavy protocol body, then runs Contract (including the drivability check) at intake and Exercise + Exhibit at every done-claim. The exempt-vs-gated decision is made by the always-on rule itself, never with the protocol unloaded.
- **Architecture**: thin trigger rule (`coding/rules/done-gate.md`, always-on) + on-fire protocol (`coding/skills/done-gate/done-gate-protocol.md`, read JIT from `{rbtv_path}`) reached through a baked skill loader (`coding/skills/done-gate/SKILL.md`). Same shape as `rbtv-orchestrating` (thin rule → skill → core-protocol). Clause-preservation map: `1-projects/rbtv-evolution/coding/done-gate-evidence/rbtv/2026-06-10-rule-split/done-gate-split-map.md`.
- **What it produces**: A drivability row per criterion at Contract time (`criterion | surface | verdict | seam built`) plus any test seams built into the feature; an owner-auditable evidence record per task — inline rows in the done message by default, or a disk sheet with capture files only when the owner requested one — and done messages that surface every `failed`, `unexercisable`, and `held-surprising` row. For orchestrated development dispatches, an independent cold verifier re-exercises the contract from day one.
- **Example**:
  > Agent at intake: "Done means: you drag the right edge of a slide element and it visibly resizes, on your real deck. Confirm?"
  > …builds, then drives the deck in a visible browser, performs the drag, captures before/after screenshots…
  > Agent: "Done — 1/1 criteria held. Exercise: dragged the right edge in a visible browser on your real deck; the element's measured width went 312→468px and held. (No file written — inline by default; ask if you want a saved sheet with screenshots.)"
  > For a criterion whose surface the agent can't drive — e.g. "you click Open…, pick a deck, and it loads" hits the native file picker — the Contract-time drivability check marks it `needs-seam` and the agent builds a test seam (env-gated path injection) while the Open… click stays a real gesture, so the done-gate exercise drives the real flow without faking the click.
