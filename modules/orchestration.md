# Orchestration

## Purpose

The long-horizon work module — creating structured plans, executing them through tiered sub-agents, and mining long sources the orchestrator can't read directly. These components make multi-session, multi-agent work repeatable: a plan written today runs without the original conversation, and execution is delegated with per-phase review instead of done in one fragile context.

---

## Components

### `rbtv-orchestrating`

- **What**: The general orchestration capability — one skill that routes any body of multi-agent work, of any work type (code included), to the right worker; dispatches each task as a self-contained artifact with a three-file state spine; verifies every return against disk (the message is a hint, the repo is the truth); and recovers from halts, worker deaths, and return drift without losing work or improvising decisions. Shaped as a thin `SKILL.md` loader → an always-loaded `core-protocol.md` (the invariants, the situation table, and the installed-model capability summary) → JIT `cards/` opened only when their situation arises. It is the single front door for orchestration: from an all-night multi-agent build down to one minimal-ceremony dispatch to a named CLI model. Model packages are doc packages it loads (see `models/` once populated), never separate skills.
  - **Core protocol** (`core-protocol.md`, always loaded): the invariants every run obeys (never execute the work yourself; no dispatch without a self-contained task artifact + state spine; disk = truth, verify every return; halts are the loop, never blind-retry; evidence must be plausible; budget question at every intake; ask-the-user taxonomy; adaptive brain floor; depth cap ≤ 2), the situation table that maps each run situation to its card, and a one-line capability summary per installed model package (with an installer-baked availability line).
  - **Cards** (`cards/`, opened JIT per the situation table): `intake.md` (entry door selection, spine init, the one pre-AFK question round — budget, code-backend, run mode), `routing.md` (boundedness → stakes → budget tree; the `(model, variant)` selector is resolved by CALLING `orchestration/models/route.py` — the card no longer hand-executes enumerate→filter→rank, but remains the algorithm authority on any script-vs-card divergence; pinned-role floors incl. Fable 5 (`claude-code-native:fable`) as the default conductor and final-plan reviewer; topology + depth cap; sdd/research/haiku leaves; batching incl. hot-zone detection against concurrent-session in-flight files), `dispatch-wrapper.md` (header+payload packaging, binding addendum incl. the conductor obligation to explicitly instruct the behavior-rule read for harnesses that do not auto-read it — codex auto-reads, kimi/qwen do not, and that read instruction must require one-file-per-call / small-batch reads, never a recursive bulk read that truncates silently mid-corpus, the unified five-field return schema, tripwire field checks — also the generated-source template rendered into each model's CLI manual; dispatches are GENERATED by the dispatch-scaffold (`orchestration/skills/orchestrating/scripts/scaffold.py`) run after four scripted pre-flight gates — package installed, manual fresh, guidance file present, output folder exists — with hand-authoring as the named fallback on a scaffold pre-flight failure plus a run-log event), `halt-recovery.md` (Halt→Decide→Resume, the L1/L2/L3 recovery ladder, doubt-escalation chain, red-sets, precondition override, USER-EXECUTED-ONLY handover), `verification.md` (return gate, review gates with pre-flagged briefs, the cold verifier at feature boundaries, debug roles, conductor-executed exit probes, and run finalization — the exit scorecard, honest completion status, and verbatim surfacing of every Human Review block), `state.md` (the three-file spine semantics, registrar discipline, propagation, the dual-write rule; the capsule regeneration discipline — every boundary write regenerates the capsule from the template skeleton (per-row liveness + volatile-facts-once), with `stamp.py --scope conductor` refusing an accreted capsule (duplicate section header, a next-dispatch claim outside Resume Point, the next dispatch stated more than once, or a Resume Point missing the three labeled bullets); state transitions route through ONE `orchestration/skills/orchestrating/scripts/stamp.py` call — worker scope writes the three worker surfaces, conductor scope adds the run-log event (the row COMPOSED from `--event-type`/`--worker`/`--outcome`, timestamp auto) + the capsule Resume Point lockstep (`Last completed`/`Next dispatch`/`Last update` labeled fields) — instead of four hand-edits, with a stamp non-zero EXIT halting the transition; a conductor-executed plan-line task with no `{id}.task.md` by design passes `--no-task-file` to skip the absent frontmatter surface and stamp the rest (refused if a task file exists, so a missing worker task file is still caught); the context-refresh offer is a deterministic checklist row of the phase-boundary procedure, not a mid-phase self-trigger).
  - **Templates** (`templates/`): `run-log-template.md` (the append-only audit log instantiated at spine init) and `state-capsule-template.md` (the mutable, atomic-overwrite resume state).
  - **Model packages** (`models/`, read JIT — not installed, resolved from `{rbtv_path}`): the capability-manifest schema (`models/manifest-schema.md`) defining the machine-readable manifest every model package ships and how the routing card consumes it (election-gated availability — `rbtv.json` `model_packages`, not raw folder presence; routes on `(model, variant)` pairs), plus its fill-in template (`models/manifest-template.yaml`). Manifests are YAML (agent-read, like the cards) — the routing card's data interface. Per-model packages (`models/{model}/` with manifest, `delta.md`, rendered manual, mirror config) populate this folder during the model-package build — **`models/kimi-code-cli/`** is the first real package (the validated bounded-code executor; it also carries the runtime Kimi contract that the M3 mandate's never-built `_shared/kimi-code-execution/` was to hold — task spec, invocation shapes, exit-75 recovery, guardrails, swarm — so a runtime orchestrator reads the package, never the human-facing kimi-cli-reference). **`models/codex-cli/`** is the second (the OpenAI Codex CLI `codex exec` worker; `evidence_status: validated` — the full headless worker contract was validated 2026-06-11 in the hypresent ui-bugfix run: codex 0.137.0 / gpt-5.5 `default` ran ~10 bounded-code dispatches end-to-end, all exit 0, disk-reconciled, stdin-hang guard fixed in rbtv@1733b17; only gpt-5.5/medium `default` exercised. (`models/` also holds build-time render/mirror test fixtures — `_fixture/` and `mirror/`'s fixture config — which are infra, not shipped packages: a directory is a model package only if it carries `manifest.yaml`.) The **mirror driver** (`models/mirror/driver/` — `guidance.py`, `library.py`, `config_assets.py`, `state.py`, `cli.py`, `__init__.py`) is a source-agnostic engine that wraps `mirror.py` and drives the full install-time mirror lifecycle for all elected packages in one pass: it reads `.claude/` rules, skills, and commands plus every `CLAUDE.md` in the workspace (never a manifest) to compose guidance files, renders each package's per-model config dir from its `mirror-assets/` seed (e.g. `models/codex-cli/mirror-assets/`, `models/kimi-code-cli/mirror-assets/`, `models/qwen-code-cli/mirror-assets/`), and copies skill loaders into `.agents/skills/`; `AGENTS.md` is shared by codex+kimi and ref-counted — deselecting one does not delete it while the other is still elected. Install integration: the full install path calls the driver after package election, and `install.py --mirror` is a mirror-only mode that refreshes artifacts for the current `rbtv.json` election without re-running component install. Driver state (managed file list, exclusions, last run timestamp) is stored in a single `model_mirror` block in `rbtv.json`, retiring the former per-script JSON state files. Three install-command arguments control mirror scope: `--exclude PATH [PATH ...]` skips paths — no guidance file is rendered beside any `CLAUDE.md` inside an excluded path; passing `--exclude` REPLACES the recorded list in `rbtv.json` (`model_mirror.excluded_paths`), omitting it PRESERVES the recorded list; applies on both a full install and `--mirror`; excluding an already-mirrored path DELETES that path's existing guidance files (banner-guarded deletion — hand-authored files are spared; never runs under `--check`). `install.py --mirror --uninstall` performs a full mirror teardown: removes ALL generated mirror artifacts for the target (guidance files, the shared `.agents/` library, and per-model config dirs) and clears the `model_mirror` block from `rbtv.json` (`--uninstall` applies only with `--mirror`).

They are NOT copied into `.claude/`; `install.py` elects packages per workspace (interactive picker when orchestration is installed, or `--model-packages kimi-code-cli,codex-cli`; omit → keep the prior `rbtv.json` selection or elect all; `""` → none), persists the choice in `rbtv.json` (`model_packages`), and bakes the resulting `Model packages installed: … — Not elected: …` availability line into the orchestrating core between the `<!-- ORCH:AVAILABILITY:BEGIN/END -->` markers so the skill knows what is routable here. The install also reconciles the workspace's `.claude/settings.local.json` permission allowlist from each package manifest's `permission_rules` (D17): an elected CLI package's entries (e.g. `Bash(kimi:*)` / `PowerShell(kimi:*)` — PREFIX rules, hence dispatch-wrapper §1's start-with-binary composition row) are added so a conductor session may spawn that worker in-session, a deselected package's entries are removed, and only the exact manifest-declared strings are ever touched — hand-added entries survive. The conditional-install surface is the manifest's `orchestration.model_packages` block (`admin/install/module-manifest.json`); the installer also runs an advisory render-freshness check that warns — never aborts — when a generated manual is stale. (One shared RBTV source serving N workspaces: last install wins the baked availability line — the same single-shared-source property the cards and rendered manuals already have.)
  - **Installer-persisted pointers** (p2-3): `rbtv.json` carries three optional pointer fields that survive re-install via `write_state` carry-forward logic: `env_file` (API-key env path), `model_plans_file` (per-model plan-overlay YAML with context-window caps and reference $/M-token data — router spec S4/S5), and `model_mirror` (mirror driver state block). Each is preserved on re-install when the caller omits the corresponding flag; `--env-file` / `--model-plans-file` flags override. The plans file schema is at `orchestration/models/model-plans-schema.md`; a tracked example at `orchestration/models/model-plans-example.yaml`. Graceful skip when absent.
  - **Deterministic router** (p2-4): `orchestration/models/route.py` — the selector: reads the elected manifests and resolves a task profile to `route {model, variant, carrier}` / `self_execute` / `halt_seam` deterministically (stdlib only; `--explain` prints the filter/rank trace; tests in `test_route.py`). A model-level script alongside the manifests it reads, like `render-manuals.py` — not an installed component. The routing card's §2a selector CALLS this script (p3-1) rather than hand-executing the enumerate→filter→rank stages; the card stays the algorithm authority (card text wins on any script-vs-card divergence, defect filed against the script).
  - **API-worker engine and API/Agent-tool packages**: `orchestration/models/_api/` is a shared infra engine — NOT a model package (no `manifest.yaml`, not electable, read JIT like `_fixture`/`mirror`). It provides a provider-agnostic runner (`run.py`, invoked as `python run.py --provider … --model … --output-folder …`) plus a sync provider base (`clients/base.py`) and per-provider chat clients (`clients/deepseek.py`, `clients/gemini.py`). The conductor routes text/synthesis leaves to non-CLI chat models through it. Three routable packages ride the engine: **`models/deepseek-api/`** (cheapest text, OpenAI-compatible JSON mode; `evidence_status: validated` — live-confirmed p6-1 2026-06-09, D-exec-15), **`models/gemini-api/`** (light web grounding; `evidence_status: validated` — live-confirmed p6-1 2026-06-09, D-exec-15), and **`models/claude-code-native/`** (Agent-tool Claude tiers opus/sonnet, `evidence_status: validated`; sibling to claude-code-cli — the in-session carrier path; no native `CLAUDE.md` load, cannot spawn sub-agents due to the nesting wall). A fourth package **`models/manus-api/`** also rides the engine as the **agentic-web worker** (`evidence_status: probe-pending`; `web_access: true`, `cost_class: high` per-task, `code_competence: none`) — it submits an autonomous task to the Manus API, polls until the server-side agent completes its own browser/tool loop, and returns a raw-dumped output file (no JSON `{files:[…]}` envelope, no code cold-verify; always `DONE_WITH_NOTES`); distinct from the chat workers, which do synchronous text synthesis. API keys resolve via the `env_file` path in `rbtv.json` (OS-env first, file fallback) — never inlined in artifacts or prompts. Per-provider price/model reference: `orchestration/docs/routing-matrix-reference.md`.
- **When to use**: Any work that needs coordinated dispatches — executing a multi-step plan through workers, running a goal end-to-end, a long-horizon/AFK build, multi-worker or cross-repo coordination, or dispatching a CLI model (kimi-code-cli, codex-cli, claude-code-cli, qwen-code-cli). Also the front door for a single standalone dispatch to a named model ("use kimi for X"). The orchestration module rule owns the discoverability trigger bar and mid-task escalation.
- **How to invoke**: Say "orchestrate this", "run this end-to-end with workers", "use kimi-code-cli/codex-cli/claude-code-cli/qwen-code-cli for X", or `rbtv-orchestrating` directly. The skill loads its core protocol and follows the situation table from there.
- **Inputs / outputs**:
  - Input: a plan (with its companion artifacts) OR a goal prompt; the run's budget answer, code-backend choice, and run mode (gathered in one pre-AFK round at intake)
  - Output: coordinated worker dispatches with per-return verification and per-boundary review/cold-verification, a three-file state spine (`run-log.md` / `state-capsule.md` / `decisions.md`) that makes the run resumable across sessions, and the delivered work itself

### `rbtv-orchestrating` (rule)

- **What**: The always-on discoverability rule for the `rbtv-orchestrating` skill (`orchestration/rules/orchestrating.md`, installed as `.claude/rules/rbtv-orchestrating.md`). It pulls orchestration-shaped work INTO the skill without the user naming it, via a balanced trigger bar: **Intake Triggers** that fire on ANY of — explicit ask, an orchestration-flagged plan, a forecast of ≥3 coordinated dispatches, multi-hour AFK intent, ≥2 worker types needed, or a **fitting single dispatch** (a self-contained unit that fits an installed + available worker beating the conductor on cost/fit — fires at minimal ceremony, not a true quick lookup); **Mid-Task Escalation** that fires while already working when context fills with work remaining, when about to dispatch the 3rd sub-agent for the same goal, or when cross-repo coordination emerges; and a **Counter-List** that NEVER fires FULL orchestration (a single dispatch needing NO worker — a direct-answer quick lookup, sub-≈30-minute tasks, and work a specialized workflow already owns — `source-mining`, wiki ingest, planning itself) so the rule is counter-shaped against eager firing (a fitting single dispatch that DOES route to a worker is the minimal-ceremony trigger above, not a counter-list exclusion). On fire, the agent announces and invokes `rbtv-orchestrating` rather than dispatching workers by hand. Enforced as a sequencing gate (invoke before any ad-hoc dispatch) plus a red-flag tripwire table against the rationalizations that skip or over-fire it.
- **When to use**: Always active once the orchestration module is installed — the agent self-detects, no user action required. It is the WHEN layer; the skill's core protocol and cards are the HOW.
- **Inputs / outputs**: Input — the task in front of the agent (at intake) and the run's growth signals (mid-task). Output — a mandatory `rbtv-orchestrating` invocation when a trigger fires; no action when only a counter-list item matches.

### `rbtv-planning`

- **What**: Interactive workflow that produces a complete, self-executing plan — a main plan file, a scope document (`decisions.md`, whose entry format carries an optional one-word `compoundable` marker as the single home for harvest-worthy findings), an artifact index (`deliverables.md`), a behavior-spec + test-plan per code feature (code-work plans), and individual micro-step task files for each action. Each micro-step file contains complete execution instructions so the plan runs without the original conversation context. It is **orchestration-aware**: a plan may declare it will be orchestrated and choose DEEP pre-resolution (resolve every foreseeable doubt with the user, emitting per-task executor/reviewer pins, allowlists, validation commands, serialization order, and a hard-halt registry in the form the orchestration router consumes) or LIGHT (resolve only critical questions, leave workers latitude) — the flag does not force orchestration, and HALT discipline is mode-independent. In DEEP mode the per-task executor pin is resolved by CALLING the router script (`orchestration/models/route.py`) — the SAME router the conductor calls at run time, so plan-time and run-time routing can never disagree — and when a pin names a model that ships a per-model contract delta, the worker-contract frontmatter is DERIVED from that delta via the dispatch-scaffold in skeleton mode rather than hand-copied. Both calls degrade gracefully on a non-orchestrated plan or a workspace without the orchestration module installed. Authoring knowledge (complexity rubric, task-file contract, spec template, dependency ordering, decisions-file discipline) is read from the shared authoring core — single source, not duplicated here. Tasks set to `human_review: required` (and every checkpoint task) instruct the executor to emit a Human Review Presentation block at completion: a list of evidence-anchored items the user should review first, plus red and yellow flags drawn from concrete criteria — never free-associated, with an explicit "None identified" affirmation when no flag triggers fire.
- **When to use**: Any multi-step task where you want a structured output you can hand off to another agent session, delegate to a teammate, or return to later. Good for projects, feature builds, operational rollouts, or anything with more than 3–4 sequential decisions.
- **How to invoke**: Say "create a plan for X" or "let's plan this out" — Claude picks up the trigger automatically. You can also say `rbtv-planning` directly.
- **Inputs / outputs**:
  - Input: task description, gathered conversationally across guided steps (incl. orchestration-flag/mode and code-work detection)
  - Output: `{plan-name}-plan.md`, `decisions.md`, `deliverables.md`, `specs/{feature}-spec.md` (code-work plans), and `phase-N/pN-X.task.md` micro-step files in a folder at the confirmed output path
- **Example**: "Create a plan for migrating our CRM to HubSpot" → Claude walks through scope, phases, and task granularity, then writes all files.

### `/rbtv-source-mining`

- **What**: Processes a long source (conversation export, transcript, book chapter, long document) that the orchestrator Claude cannot read directly due to context limits. It chunks the source, dispatches sub-agents to extract decisions or concepts from each chunk, groups the results, and synthesizes them into either a **reconciled document** (updates an existing doc with session decisions + user line-comments) or a **study note**. The orchestrator never reads the source — only sub-agents do.
- **When to use**: You have a long conversation or document you want to mine for decisions, incorporate into an existing spec, or turn into study notes. Especially useful after long planning sessions where you want to update a PRD or plan with what was decided.
- **How to invoke**: `/rbtv-source-mining` — Claude asks for source path, mode (reconcile or study), and target document.
- **Inputs / outputs**:
  - Input: source file path, mode selection, target document(s) for reconcile or destination for study
  - Output (reconcile): target document(s) updated in place + a delta document per target showing what changed
  - Output (study): new study note at confirmed destination
- **Example**: `/rbtv-source-mining` → "reconcile" → point to a long strategy conversation → point to your current strategy doc → Claude mines the conversation and updates the doc with decisions made, showing a diff.

## Shared Components

### Shared Authoring Core

- **What**: A single source of artifact-authoring knowledge at `orchestration/workflows/_shared/authoring/`, consumed by BOTH the interactive `rbtv-planning` workflow and orchestration intake writer agents so both doors produce identical artifact contracts. It owns: the task-file contract (zero-context self-containedness, file-operation verbs, allowlists, context budgets, owner-observable acceptance criteria, and the per-model contract plug-in seam), a behavior-spec + embedded test-plan template for code work, a complexity rubric (Dependencies and Decision Density widened beyond the 1–3 scale to match observed coordination/decision load) that routes a body of work to its entry door, dependency-ordering rules with shared-file serialization, and the worker-facing `decisions.md` entry discipline (decision/rationale/scope only; append-don't-rewrite; size floor on rewrites).
- **When to use**: Read by planning and intake when authoring task files, specs, and dependency-ordered plans — not invoked directly by users. It is authoring KNOWLEDGE (what an artifact must contain, how to size/order/discipline it), distinct from workflow MECHANICS (step sequencing, plan-document structure), which stay in the consuming workflow.
- **Inputs / outputs**: Input — a body of work to author artifacts for. Output — task files, specs, and orderings that satisfy one shared contract regardless of which workflow or which executor model produced or runs them.

## How They Fit Together

`rbtv-orchestrating` is the general front door for long-horizon, multi-agent work — it ingests a plan or a goal, routes each task to the right worker, and gates every return; the `rbtv-orchestrating` rule is the discoverability layer that pulls qualifying work through that front door (intake trigger bar + mid-task escalation + counter-list) without the user naming it; `rbtv-planning` writes the plans it ingests. `/rbtv-source-mining` mines the long conversations that planning and execution produce, folding decisions back into the plan or PRD.
