## 5. ASSEMBLE.PY

### CLI interface (verbatim argparse setup)

```python
parser = argparse.ArgumentParser(
    description="Tecer slide-library assembly mechanics (stdlib-only).",
    formatter_class=argparse.RawDescriptionHelpFormatter,
    allow_abbrev=False,
)
mode = parser.add_mutually_exclusive_group(required=True)
mode.add_argument("--preset", metavar="NAME", help="assemble from a named preset")
mode.add_argument("--slides", metavar="id1,id2,...",
                  help="assemble from an explicit comma-separated id order")
mode.add_argument("--check", metavar="FILE",
                  help="scan a file for leftover {{TOKEN}}s (exit 1 if any)")
mode.add_argument("--catalog", action="store_true",
                  help="regenerate catalog.html from the manifest")

parser.add_argument("--out", metavar="FILE", help="output deck path (assembly modes)")
parser.add_argument("--client-logo", metavar="PATH", help="path to the client logo for {client-logo}")
parser.add_argument("--accent", metavar="#HEX", help="client accent color injected as --client-accent")
parser.add_argument("--title", metavar="T", help="document title (overrides preset default)")
parser.add_argument("--lang", metavar="L", help="document language (overrides preset default)")
```

**`allow_abbrev=False`** — stale abbreviated flags (e.g., `--client`) error loudly (exit 2) instead of prefix-matching.

**Removed flags (documented in CLAUDE.md p5-1):** `--client` and `--date` — had no mechanical effect; removed to prevent silent misuse.

### Feature inventory

| Feature | Implementation detail |
|---------|----------------------|
| **Four modes** | `--preset NAME --out FILE`, `--slides id1,id2,... --out FILE`, `--check FILE`, `--catalog` — mutually exclusive group |
| **Preset resolution** | Reads `presets.md` → regex-finds fenced YAML blocks → minimal YAML parser (`_parse_yaml_block`) that handles multi-line `[...]` slides lists |
| **Explicit order** | `--slides` comma-split; each id looked up in manifest; unknown id → `die()` (exit 1) |
| **Manifest parsing** | `parse_manifest()` — strict 9-column check against `MANIFEST_COLUMNS`; separator rows skipped; column-count mismatch → `die()` |
| **Theme inlining** | `theme.css` read and inlined at `/* {{THEME_CSS}} */` marker in `base.html` — assembled deck is fully standalone |
| **Slide renumbering** | `renumber_slides()` — regex replaces `<div class="slide-number">{{N}}</div>` sequentially; counter only advances on real token; cover/opening slides without the token are skipped |
| **Lang injection** | `--lang` (or preset default) fills `{{LANG}}` in `<html lang="...">` |
| **Title injection** | `--title` (or preset name / "Tecer") fills `{{TITLE}}` via `html.escape()` |
| **Accent injection** | `--accent "#HEX"` → inserts `:root { --client-accent: #HEX; }` at `/* {{ACCENT_CSS}} */` marker; if omitted, the whole accent line is removed |
| **Asset resolution — two-root scheme** | `resolve_asset_source()`: entry contains `/` → resolves from `REPO_ROOT` (tecer-biz/); bare filename → resolves from `slide-library/assets/`; `{client-logo}` → from `--client-logo` flag |
| **Asset copy** | All assets resolved up front (validates existence before any write); then copied to sibling `assets/` of `--out`; uses `shutil.copy2` |
| **Unfilled token report** | After assembly, `TOKEN_RE.findall(document)` scans output; all remaining `{{...}}` printed as the Step-5 to-do list |
| **`--check` mode** | Reads a file, lists unfilled tokens, exits 1 if any remain |
| **`--catalog` mode** | Reads all manifest rows in order; prepends a label bar (`id · kind · audience · lang · summary`); renumbers; inlines theme; writes `catalog.html` — no client accent |
| **Error handling** | `die()` function — prints "ERROR: ..." to stderr and `sys.exit(1)`. Fails loud on: missing manifest/presets/base.html/theme.css, unknown slide id, missing fragment file, missing asset, malformed manifest row, malformed preset YAML, empty `--slides`, missing `--out` in assembly mode |
| **Validation-before-write** | All fragment/asset resolution happens before any file is written — no partial outputs on failure |
| **Python version** | No version check; uses only stdlib: `argparse`, `html`, `re`, `shutil`, `sys`, `pathlib.Path` — **stdlib-only** |
| **Code size** | 440 lines |

## 7. THEME + BASE — Composition Model

### base.html skeleton

```html
<!DOCTYPE html>
<html lang="{{LANG}}">
<head>
<meta charset="UTF-8">
<meta name="viewport" content="width=device-width, initial-scale=1.0">
<title>{{TITLE}}</title>
<link rel="preconnect" href="https://fonts.googleapis.com">
<link href="https://fonts.googleapis.com/css2?family=Outfit:wght@300;400;500;600;700;800;900&display=swap" rel="stylesheet">
<link href="https://fonts.googleapis.com/css2?family=Space+Mono:wght@400;700&display=swap" rel="stylesheet">
<link rel="stylesheet" href="https://cdnjs.cloudflare.com/ajax/libs/font-awesome/6.5.1/css/all.min.css">
<style>
/* {{ACCENT_CSS}} */
/* {{THEME_CSS}} */
</style>
</head>
<body>
<!-- {{SLIDES}} -->
</body>
</html>
```

**5 markers in base.html:**
1. `{{LANG}}` — HTML lang attribute (`pt` / `en`)
2. `{{TITLE}}` — document `<title>` (html-escaped)
3. `/* {{ACCENT_CSS}} */` — optional `:root { --client-accent: #HEX; }` injection
4. `/* {{THEME_CSS}} */` — full `theme.css` content inlined (entire design system)
5. `<!-- {{SLIDES}} -->` — concatenated fragment HTML

**External dependencies (NOT inlined):** Google Fonts (Outfit, Space Mono) + FontAwesome 6.5.1 via CDN. The assembled deck is NOT fully offline-capable unless CDN resources are cached.

### How a final deck is composed

1. `base.html` is the skeleton
2. `theme.css` (~46KB) is inlined verbatim inside `<style>` at the `/* {{THEME_CSS}} */` marker
3. Optional `--accent` CSS var injected before theme
4. Fragment HTML for each slide is concatenated in order and substituted at `<!-- {{SLIDES}} -->`
5. Slide numbers renumbered
6. Assets copied to sibling `assets/` directory beside the output file
7. Result: single self-contained `.html` file with all CSS inline; assets as relative `assets/` references

### Standalone rendering requirements for a slide fragment

A fragment is NOT standalone — it depends on:
- `theme.css` (all component classes defined there)
- Google Fonts CDN (Outfit, Space Mono)
- FontAwesome CDN (icon classes `fa-solid`, `fa-brands`)
- `assets/` directory at a known relative path (images referenced as `assets/{filename}`)
- `base.html` for the document wrapper

## ORCHESTRATOR NOTES

The 5 facts most likely to shape the convention spec:

1. **The manifest's 9-column schema is the authoritative metadata contract.** Any RBTV-generic convention must either adopt these columns verbatim or define a strict superset. The `kind` field (ready vs template) is the single most important field for an assembly engine — it determines whether tokens must be filled and whether the slide requires a creative pass. This distinction belongs at the convention level, not just in Tecer's manifest.

2. **Fragment = bare `<section>`; all CSS lives in `theme.css`; all metadata lives in `manifest.md`.** This clean separation is the convention's architectural invariant. A generic library MUST enforce: no `<style>` in fragments, no in-file metadata headers, no asset embeds except relative `assets/` references. Violation of any of these breaks the assembler's inlining model.

3. **The two-root asset resolution scheme (`/` = repo root; bare = library assets) is a load-bearing design decision** (called "Decision E" in CLAUDE.md). A generic convention must either adopt this or replace it with an explicit `@repo` vs `@library` prefix. The current implicit convention is fragile — a bare filename that accidentally contains `/` (e.g., versioned paths) would silently resolve from the wrong root.

4. **All 11 as-built entries are retroactive; no entry was written during original deck assembly.** The upstream rule (Step 8) and as-built log are aspirational — the log exists but represents a retroactive record, not a live discipline. A generic convention must decide: is the as-built log a mandatory gate (blocking deck delivery until the entry exists) or an optional record? Tecer's current answer is effectively optional in practice (all entries are "n/a (retro)"). An assembly engine could enforce this as a post-assembly hook.

5. **The `{{TOKEN}}` + HTML-comment-as-spec pattern is the core creative pass interface.** Each token's comment is the authoring spec for that slot. A generic convention should formalize this: the comment IS a machine-readable prompt to the filling agent, not documentation. A spec format for these comments (required fields: content type, depth, example, counterparty anchoring instruction) would make the creative pass more automatable and more consistent across library maintainers.
