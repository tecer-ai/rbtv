# Task T4: Parent ↔ Iframe Bridge

## Goal

CREATE the parent↔iframe protocol: a parent-side command sender + event receiver and an iframe-side command dispatcher + event emitter, origin-filtered and `source:'hyp'`-tagged.

## Execution Flow

### Phase: Understand
1. MUST read every Context File.
2. Mark T4 in progress in BOTH locations.
3. Confirm the exact command/event names from `01` §3 — downstream feature tasks depend on these signatures.

### Phase: Execute
1. CREATE `app/js/bridge/bridge-parent.js`: `command(type,payload)→Promise<result>` (resolves on the matching ack), `on(type,handler)`; reject messages where `event.origin !== location.origin` or `data.source !== 'hyp'`.
2. CREATE `runtime/js/bridge-iframe.js`: `command(type,payload)` dispatch table (initially with a `noop` + `ping` for testing; feature tasks register real handlers), `emit(type,payload)` → `window.parent.postMessage({source:'hyp',type,payload}, location.origin)`.
3. Define the registration mechanism feature modules use to add command handlers and emit events.

**Discovery Handling:** simple → resolve + shape.md; complex → add task, notify.

### Phase: Validate
1. Parent sends `ping`; receives ack.
2. Runtime emits a test event; the parent `on` handler fires with the payload.
3. A cross-origin or non-`hyp` message is ignored.

### Phase: Close
1. Append decision/discovery to shape.md (standalone) or defer (orchestration).
2. Mark T4 complete in BOTH locations after acceptance passes.
