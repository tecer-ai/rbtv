## Original Shaping (Planning Phase)

### Scope Definition

**What this plan accomplishes:**
- Align RBTV skill YAML frontmatter with Anthropic's official Claude Skills specification
- Fix `name` field mismatch across all 7 existing skills (name must match folder name)
- Extend the skill template to document optional Anthropic fields (license, compatibility, metadata, allowed-tools)
- Create a dedicated Cursor rule governing skill structure standards
- Archive the Anthropic "Complete Guide to Building Skills for Claude" as platform knowledge
- Update the builder agent (fernando.md) to reference the new skill-standards rule

**What this plan does NOT include:**
- Retrofitting optional fields (compatibility, metadata) into existing skills — these are optional and added incrementally
- Changing the installer (`install-rbtv.py`) — it copies skills as-is, no frontmatter parsing
- Changing the tools-manifest format — it uses `id` (short form), not the YAML `name` field
- Updating admin-installed copies in `.cursor/skills/` — regenerated by installer on next run

### Key Decisions

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Name field alignment | Change `name` to match folder name (e.g., `doc` → `bmad-rbtv-doc`) | Anthropic spec requires name == folder name; user confirmed |
| Optional fields approach | Document in template, don't retrofit | Keeps existing skills working; fields added incrementally when relevant |
| Rule scope | Create new `bmad-rbtv-skill-standards.mdc` in `_config/.cursor/rules/` | No existing rule covers skill structure; canonical source is `_config/` |
| Archive format | Platform Knowledge document in prompting-assistance data | Matches existing pattern (claude_projects, chatgpt_projects, etc.) |
| Thin loader convention | Skills remain thin loaders (20-35 lines) by default | RBTV skills delegate to agents/workflows/tasks; optional subdirs documented but not required |

### Constraints

| Constraint | Source | Impact |
|------------|--------|--------|
| Skills are thin loaders | RBTV architecture | Skills only load and delegate; no logic inside SKILL.md |
| `_config/` is canonical source | RBTV installer design | All skill edits go to `_config/.cursor/skills/`, not `.cursor/skills/` |
| tools-manifest uses `id` not `name` | Existing manifest format | Changing `name` in YAML does not affect manifest; no manifest changes needed |
| Installer copies skills as-is | `install-rbtv.py` behavior | No installer changes needed; next run picks up new files |

## Standards Applied

### RBTV Component Standards

| Standard | Application in This Plan |
|----------|-------------------------|
| Atomic files | All new files must be self-contained, no content repetition |
| Thin loader pattern | Skills remain thin loaders; template documents but doesn't enforce optional subdirs |
| Canonical source in `_config/` | All skill and rule edits target `_config/.cursor/` paths |
| tools-manifest registration | New skill must be added to tools-manifest.csv |

### Plan-Specific Rules

| Rule | Enforcement |
|------|-------------|
| Edit `_config/` not `.cursor/` | All skill/rule changes go to `_config/.cursor/`; installer propagates |
| Preserve existing skill behavior | Name change must not alter skill triggering or activation logic |
| Platform knowledge format | Archive must follow existing pattern in `workflows/prompting-assistance/data/platform_knowledge/` |

### Tool Mode Selection

| Scenario | Mode | Rationale |
|----------|------|-----------|
| Need prior context | Skill | Preserves conversation history |
| Context saturated | Subagent | Fresh context window |
| Complex validation | Subagent | quality-review needs focused evaluation |
| Quick lookup | Skill | Minimal overhead |
| Already in subagent | Skill only | Subagents cannot nest |
