## F1 — `e.dist`/`e.width` DOUBLE under `fixedDirection:[0,0]`; spec S3-6 and R12-1 are both wrong — **BLOCKER**

**The spec's central R12 claim is false at the source.** S3-6 states: *"with `fixedDirection:[0,0]` Moveable reports the SAME `e.dist[axis]` for the dragged edge, and the S3-1 write path (`target = beforeRect[axis] + e.dist[axis]`) applies identically."* The vendored compute proves the opposite — **`e.dist[0]` is exactly doubled in center mode.**

**Evidence (extracted from `moveable.min.js`).** The resize distance is computed in `Ra(t, e, n, r)` (offset 150863; `t` = dragged direction `X`, `n.fixedDirection` = `g`):

```js
var g = n.fixedDirection,
    m = [0,1].map(e => Ia(t[e] - g[e])),        // per-axis |direction − fixedDirection|
    b = [0,1].map(t => { var e = m[t]; return 0 !== e && (e = 2/e), e; });   // b = 2/m
...
D = br({datas:n, distX:l, distY:f});            // raw pointer delta projected on direction
D = b.map((t,e) => D[e]*t);                      // pointer delta × b
u = Oa([a,s], D, o, t, e)[0];                     // → distWidth
return { distWidth:u, distHeight:c };
```

`distWidth` becomes `e.dist[0]` and feeds `rt = startWidth + distWidth` = `e.width` (resize compute @101000–102600, confirmed verbatim: `W = U - D`, `rt = g + W`, `dist:[W,j]`).

Trace the multiplier `b[0]` for an E-handle (dragged direction X = `[1, …]`):
- **Default one-sided** (`fixedDirection` = opposite edge = `[-1, …]`): `m[0] = |1 − (−1)| = 2` → `b[0] = 2/2 = 1`. dist = pointer delta × 1.
- **Center** (`fixedDirection = [0,0]`, set by R12 for Alt): `m[0] = |1 − 0| = 1` → `b[0] = 2/1 = **2**`. **dist = pointer delta × 2.**

So for a +Δ pointer drag in center mode, Moveable reports `e.dist[0] = 2Δ` and `e.width = startWidth + 2Δ`. The box grows by **2Δ** (Δ per side) and the **dragged edge tracks the cursor 1:1** — that is correct, standard center-resize UX (PowerPoint behaviour), and it is what Moveable's native frame already produces for the healthy/absolute/grid paths.

**Two concrete failures this creates:**

1. **R12-1 contract is inverted.** spec.md line 188 (R12-1) and test-plan E-R12-1 (lines 116–117) assert the box grows by **Δ TOTAL** (each edge ±Δ/2, dragged edge drifts Δ/2 *behind* the cursor). Against real Moveable, a +60 Alt E-drag on the healthy `twin` (block path, consumes `e.width` directly) produces `e.width = start + 120` → width **+120**, each edge **±60**. The test asserts `after["w"] − before["w"] ≈ 60 (delta=3)` and `right − before ≈ 30`; reality is **+120 / +60**. **E-R12-1 and E-R12-2's symmetry math, E-R12-4, and the R12-1 contract all fail or encode wrong behaviour.**

2. **S3-6 "composes unchanged" breaks R10 under Alt.** The R10 write is `flexBasis = beforeRect.width + dist[0]`. In center mode `dist[0] = 2Δ`, so the basis (and rendered width on a grow-neutralised item) grows by **2Δ for a Δ cursor move** — re-introducing a 2× amplification *only when Alt is held*. E-R12-4 asserts width `+60 (delta=3)` for a +60 drag on the accent; reality is `+120`. The R10 dist-write does NOT compose unchanged with center anchor — it needs the symmetric case handled explicitly.

**Why the diagnosis didn't catch it:** the diagnosis (R12 section) verified `startFixedDirection` snapshotting and that `fixedDirection:[0,0]` pins the center — both TRUE and confirmed here (offset 99992). It never traced the `b = 2/m` distance scaling, so it (correctly) said "`e.dist` arrives already correct for that anchor" meaning *the dragged edge tracks the cursor* — which is right. The spec then mis-paraphrased "correct for the anchor" into "the same numeric `dist`," which is the error.

**Correct contract (the dragged edge tracks the cursor 1:1, both edges mirror):**

- R12-1 (rewrite spec line 188): *"an E-handle drag of +Δ with Alt held moves the dragged (right) edge to track the cursor 1:1 (`|ΔrightEdge − Δcursor| ≤ 3`) AND mirrors the left edge by the same Δ (`|ΔleftEdge − Δcursor| ≤ 3`); total width grows by **≈2Δ** (`|Δwidth − 2Δcursor| ≤ 3`)."* This is the honest, native behaviour — and it is the behaviour D1 actually wants ("dragged edge tracks the cursor 1:1; Alt = symmetric from center").
- R12-4 (test-plan lines 134–137): assert `after["right"] − before["right"] ≈ 60 (delta=3)` (dragged edge 1:1, NOT 30) and `after["w"] − before["w"] ≈ 120 (delta=3)` (NOT 60). Left edge `before["left"] − after["left"] ≈ 60`.
- E-R12-1 (test-plan lines 116–118): `before["left"] − after["left"] ≈ 60 (delta=3)`, `after["right"] − before["right"] ≈ 60 (delta=3)`, `after["w"] − before["w"] ≈ 120 (delta=4)`.
- S3-6 (spec line 41): replace "Moveable reports the SAME `e.dist[axis]`" with: *"Moveable doubles `e.dist[axis]` under center anchor (`b = 2/|dir−fixedDir|` in `Ra`); the R10 write therefore grows the box by `2·Δcursor` (Δ per side), keeping the dragged edge 1:1. The R10 write path needs NO special math because `beforeRect+dist` with the doubled dist already lands the dragged edge under the cursor — but the spec must state the width grows 2Δ, not Δ, so tests assert the right magnitude."*

**Net:** the FIX (set `fixedDirection:[0,0]`) is correct and needs no code change beyond what S3-5 already specifies. Only the **contracts and all R12 test tolerances** are wrong (they encode a Δ-total / Δ/2-drift model that contradicts both the source and D1). This MUST be corrected before implementation or the implementer will either (a) write passing tests that lock in cursor-lagging behaviour the user explicitly rejected, or (b) add compensation math to force width=Δ, violating D1 and de-tracking the cursor.

## F2 — W-handle `dist[0]` sign: spec claim is CORRECT (confirm) — no change

The spec (line 149) claims `e.dist[0]` for a W-handle is "already signed for the left-edge drag … so `beforeRect.width + dist[0]` is correct without special-casing." **Verified.** `e.dist[0] = W = U − D` where `U` = current boundingWidth, `D` = start boundingWidth. `dist` is a **SIZE delta of the bounding box**, not a cursor delta: dragging the W (left) edge leftward enlarges the box, so `U > D` and `W > 0`; `beforeRect.width + W` is the new width. The default `fixedDirection` for a W-drag holds the east edge (`b[0] = 2/|−1 − 1| = 1`, magnitude 1:1). `beforeRect.width + dist[0]` needs no handle special-casing. **F2 resolves in the spec's favour — keep as written.** (One caveat folds into F1: under Alt on a W-handle, `b[0] = 2/|−1 − 0| = 2`, dist doubles, same as F1.)

## F3 — R14 stamping idempotence: holds, with ONE residual hole — **MAJOR**

I walked each path through the specced serializer pass against `serializer.js` + `comments.js`:

- **Multiple consecutive saves / reopen-resave:** SAFE. `serialize()` deep-clones the live DOM (line 253). A reopened file's prior `data-hyp-agent` is a plain `data-hyp-` attribute on the clone → removed by `stripClone` Phase B (lines 178–184, `attr.name.startsWith("data-hyp-")`) → re-stamped from a freshly-derived `agentStampMap`. No accumulation. E-R14-6 is valid.
- **Resolve-then-save / delete-then-save:** SAFE. `agentStampMap` (S3-13) and `buildAgentBlock` both iterate `agentInstruction===true && resolved!==true`; resolved/deleted threads contribute nothing. No stale id. E-R14-3 valid.
- **Two threads on one element:** SAFE. `agentStampMap` returns `Map<Element,string[]>`; `ids.join(" ")` → `data-hyp-agent="id0 id1"`; `[data-hyp-agent~="id0"]` token-matches. E-R14-2 valid.

**The hole — `resolveCloneNode` is fragile against the very mutation R14 exists to survive, but more importantly its child-index walk can desync from the freshly-inserted island.** `resolveCloneNode` (serializer.js:227–247) maps live→clone by **`childNodes` index** (text nodes included). The stamping pass (S3-13) runs *after* `stripClone` removed nodes from the clone. `resolveCloneNode` walks `cloneRoot` (the post-strip clone) using indices computed from the **LIVE** DOM (`Array.from(parent.childNodes).indexOf(cur)`). For any agent-anchored element that has a `hyp-`stripped element or stripped comment-marker as a preceding sibling **at any ancestor level**, the live child-index no longer matches the clone child-index → `resolveCloneNode` returns the wrong node or null → stamp lands on the wrong element or is silently dropped.

This is not hypothetical for this codebase: comment **markers** are `hyp-comment-marker` `<div>`s appended to `document.body` (comments.js:321) and the interaction **wrapper** (`hyp-interaction-wrapper`) is appended to body (interaction.js:185). Both are stripped from the clone. Any agent-tagged element that is a later body-child than a marker/wrapper (i.e. essentially any of them, since markers append to body) shifts. The existing contenteditable sweep (serializer.js:209–218) uses the same `resolveCloneNode` and "works" only because it runs BEFORE the island/agent-block insertion and tolerates an occasional miss (a stray contenteditable left over is cosmetic); a **missed or mis-targeted agent stamp is a correctness failure** (the head block's `target:` selector then resolves zero or the wrong element — exactly the D4 failure R14 exists to fix).

**Concrete failure:** open `agent-comments.html`, agent-tag `#p-arch` (in the 2nd section). At save time, body has appended `hyp-comment-marker` + `hyp-interaction-wrapper` children (stripped from clone). `resolveCloneNode` walks from live `#p-arch` up to `documentElement`; at the `body` level the live index of `<section id="sec-arch">` includes no extra offset *yet* (markers are appended AFTER the sections, so sections keep their indices) — **but** if the user has selected an element (wrapper present) AND any marker was inserted before a section via `insertBefore`-style logic, or the doc has body-level whitespace/comments, the index drifts. The risk is real and unguarded; **no test exercises a stamp on a non-first element with live-only siblings present.**

**Fix:**
1. Spec S3-13 / change-map (spec line 365): **stamp the LIVE element first with a temporary marker, OR map via a strip-stable key.** Cleanest: in `agentStampMap`, instead of returning live Elements, have the serializer stamp by **re-running `matchAnchor` semantics on the CLONE** — but `matchAnchor` queries `document`, not the clone. Therefore the implementable fix is: **temporarily set `data-hyp-agent` on the LIVE element inside `serialize()` AFTER cloning is impossible** — no. The correct, minimal fix: **stamp the live element with `data-hyp-agent` BEFORE the clone is taken, exempt `data-hyp-agent` from the strip for this one pass, then remove it from the LIVE DOM in a `finally`.** This is diagnosis "approach 1 applied transiently" and sidesteps `resolveCloneNode` entirely. If the spec insists on clone-side (approach 2), it MUST replace the index-based `resolveCloneNode` with an element-identity map built **at clone time** (e.g. tag both live and clone via a temporary `data-hyp-clone-idx` sequence number before stripping), because index-walking a post-strip clone with live indices is unsound.
2. Test-plan: ADD **E-R14-8** — agent-tag a NON-FIRST element (`#p-arch`, `#li-2`) **with a selection active** (so the wrapper + markers are in the live body), save, and assert the `target:` selector resolves that exact element on reopen. This is the case that exposes the desync; current E-R14-1/-2/-5 all tag `#p-lead` (first-ish) and may pass by accident.

## F5 — R11 equal-size snap × R10 flex write: feedback-loop risk — **MAJOR**

Traced the override interaction:

- **Snapped value into flex-basis:** S3-8 says the override replaces the target `t` (= `beforeRect+dist`) with the matched candidate value, and "the S3-1/S3-2 write uses the matched value." So `flexBasis = matchedWidth + "px"` with `flexGrow:0`. Consistent — the snapped width flows into basis correctly on a grow-neutralised item. OK.

- **Feedback loop on subsequent frames — REAL.** `sizeCandidates` are cached at `resizeStart` from `getElementGuidelines(el)` rects (S3-7). The candidate set INCLUDES siblings. On `flow-grow.html`, the accent's siblings are `node-a`/`node-b` (`flex-grow:1, flex-basis:0%`). When R10 neutralises the accent's grow and sets an explicit basis, the row reflows and **the siblings' rendered widths change** (they absorb/release the freed grow space). The cache is taken at start so the cached sibling widths are pre-gesture — but the **`twin` is the intended match target and it does NOT reflow** (it's in a separate block, S3-7 fixture). So the headline E-R11-2 (snap to `twin`) is stable. HOWEVER: the candidate set also contains `node-a`/`node-b`, whose **pre-gesture** widths (~? px each, grow-shared) are arbitrary values now frozen as snap targets. As the user drags the accent through one of those frozen widths, the accent snaps to a sibling width that **no longer exists** (the sibling reflowed). This produces a phantom snap to a stale value mid-gesture — visible as a stutter and a wrong landed size. Spec calls this "accepted" (edge case "Sibling reflow … cached dims are the intended match target") but on a flex-grow row the siblings' *start* widths are NOT a meaningful target — they were grow-derived, not author-set. **Snapping to a sibling's transient grow width is a mis-feature, not a documented-acceptable bound.**

- **4px threshold re-entry:** once snapped (`t := cand`), the NEXT frame recomputes `t = beforeRect + dist` fresh (dist is cumulative from start, not relative to the snapped frame), re-tests `|t − cand| ≤ 4`, and re-snaps if still in band. No cumulative drift / no oscillation, because the override does not feed back into `dist` (dist comes from Moveable's pointer math, untouched by our basis write). **Within the 4px band the width is pinned to `cand`; leaving the band releases to 1:1.** This part is sound — no runaway loop. Good.

- **resizeEnd with a snapped final frame:** the last `onResize` wrote `flexBasis = cand`. `onResizeEnd` (interaction.js:202–211) re-captures `after = captureSizingState` from the live inline style → `after.flexBasis = cand+"px"`, `after.flexGrow = "0"`. The command's `after` map is the snapped value. Undo restores pre-gesture. **Consistent — the snapped value is what's committed and what serializes.** OK. E-R11-2's `assertAlmostEqual(accW1, twinW, delta=0.5)` is implementable.

**Fix:** restrict R11 equal-size candidates to elements whose width/height is **author-explicit**, OR exclude same-row flex-grow siblings of the resized element from the equal-size candidate set (they're already covered by *position* guides). Minimal spec change (S3-7, spec line 42): *"the equal-size candidate set excludes flex-grow siblings sharing the resized element's flex parent (their rendered size is solver-derived, not a meaningful match target); `twin`-style out-of-flow or explicitly-sized peers remain."* Add a guard test **E-R11-5**: drag the accent slowly through a frozen sibling start-width and assert it does NOT snap there (only `twin` snaps). Without this, E-R11-2 may pass while real decks mis-snap to phantom sibling widths.

## F6 — Fixture fidelity & red-first proof: the fixture is faithful, but red-first is NOT GUARANTEED for the dead-zone tests — **MAJOR**

**Geometry fidelity:** `flow-grow.html` (spec lines 80–101) faithfully reproduces G0: `flex-grow:1.4` accent, `justify-content:center` parent, `flex-grow:1; flex-basis:0%` siblings, arrows omitted (the real deck's `flow-arrow{grow:0}` are inert spacers — their absence doesn't change the accent's grow-share math). `width:1200px` parent vs the live 1471px is immaterial to the *ratio* mechanism. The amplification (slack) regime WILL manifest pre-fix: the accent has slack (1200px row, 3 grow items, accent grows 1.4-share), so the current `flexBasis = e.width` write re-inflates. **E-R10-1 goes red on current master. Good.**

**The dead-zone tests may NOT go red on this fixture — red-first proof is unverified.** The live dead zone (R10b) required the accent to be the **dominant grower consuming ALL container leftovers** (`grow:1.4` vs siblings, with the accent's basis already at 1323px inline so rendered = leftovers). On the PRISTINE `flow-grow.html`, all three items start at `flex-basis:0%` and the accent's `grow:1.4` vs two `grow:1` gives it only `1.4/3.4 ≈ 41%` of the row — it is **NOT** consuming all leftovers, so there IS slack in both directions and the FIRST gesture is the *amplification* regime, not the dead zone. **E-R10-2's premise** ("after the first gesture establishes grow:0, a second drag shrinks 1:1") tests the POST-fix path and only goes red against a *broken fix*, not against current master — but **E-R10-2 on current master:** the first gesture (E-R10-1's grow) on master writes `flexBasis = e.width` (no grow:0), so the second gesture re-enters grow and the dead-zone IS reproduced on master for gesture 2. So E-R10-2 likely is red on master, but for a *different reason* than the contract states (the spec frames it as "no longer pinned to container leftovers"; on master after a basis-inflating first gesture, the accent may or may not be leftover-pinned depending on the inflated basis). **The red-first behaviour of E-R10-2 / E-R10-2b is unproven and fixture-dependent.**

More concretely: **E-R10-2b** ("first gesture −100 shrinks 1:1, delta=3") on the pristine fixture — on CURRENT master, dragging the accent's E-handle −100 with slack present writes a *smaller* `flexBasis`, the grow solver refills part of it, rendered width shrinks by some fraction <100 (amplification-in-reverse, not a clean dead zone). It will likely fail `delta=3` on master (red — good) but NOT because of a "dead zone"; the test's *falsification rationale* (line 67/72) is mis-stated.

**Fix:**
1. To reproduce the TRUE dead zone (leftover-pinned), the fixture needs the accent to dominate. Either give the accent a much larger grow (`flex-grow:8` vs siblings `1`) so it consumes ~80% and the row has near-zero slack for it, OR seed an inline `flex-basis` near the row width (the live file had `flex-basis:1323px`). The spec's "PRISTINE-equivalent, no inline basis residue" (line 78) is precisely what REMOVES the dead-zone condition. **Recommend: add a second fixture variant `flow-grow-deadzone.html` with the accent at `flex-basis` ≈ full row (mirroring the live `1323px` residue) for E-R10-2/-2b, OR raise the accent's grow so leftovers vanish.** Otherwise the dead-zone tests are not red-first proofs of the dead-zone bug.
2. Run each new RED test against current master BEFORE writing the fix and record the actual failure mode in the test-plan (the spec asserts red-first but provides no evidence the dead-zone subset actually fails on master with THIS fixture). This is a process gap: "Would the specced tests FAIL against current master" is **assumed, not demonstrated**, for E-R10-2/-2b/-6.

### G4 — `editComment`/`editReply` undo "delete editedAt if absent" needs explicit absence capture — **MINOR**
S3-11 / change-map (spec line 302) says undo restores `editedAt` "or its absence." `before = {body, editedAt}` where `thread.editedAt` is `undefined` pre-first-edit. Undo must `delete thread.editedAt` when `before.editedAt === undefined`, NOT set it to `undefined` (a thread with `editedAt: undefined` serializes differently — `writeIsland` (comments.js:263–273) does NOT currently list `editedAt`, so it would be dropped anyway, but `toJson` (line 389) also omits it). **The spec (line 306) says ADD `editedAt: t.editedAt` to writeIsland AND toJson.** Once added, `editedAt: undefined` will serialize as a present key with `null` (JSON.stringify drops `undefined` in objects → key omitted; OK). E-R13-2 asserts "NO `editedAt` key" post-undo — `JSON.stringify({editedAt: undefined})` → `{}`, key absent. So setting `thread.editedAt = undefined` on undo HAPPENS to satisfy the island assertion, but the IN-MEMORY object has the key. **Fix:** change-map must say `if (before.editedAt === undefined) delete thread.editedAt; else thread.editedAt = before.editedAt;` to keep the in-memory store clean (E-R13-2 reads the island so it'd pass either way, but a later `toJson` consumer could see the stale key). Minor.

### G5 — S3-9 hint overlay appended to `wrapper`, but `wrapper` is `pointer-events:none` and z-index 999998 — same as markers — **MINOR**
S3-9 (spec line 250) appends `hyp-size-hint` to `wrapper`. The wrapper is `position:absolute; pointer-events:none` (interaction.js:176–184). A child with `pointer-events:none` inherits fine (E-R11-3 asserts `pointer-events:none` — OK). But the wrapper's z-index (999998) equals the comment marker z-index and the Moveable control box sits inside it; an absolutely-positioned hint child using `matchEl.getBoundingClientRect()` (viewport coords) appended to an `position:absolute; top:0; left:0` wrapper that spans `scrollWidth×scrollHeight` will be MIS-positioned if the page is scrolled (getBoundingClientRect is viewport-relative; the wrapper origin is document-relative). The comment markers handle this by adding `scrollX/scrollY` (comments.js:283–284). **Fix:** S3-9 must specify the hint position adds `window.scrollX/scrollY` to the matched rect (mirror `positionMarker`), or append to `document.body` like markers do. As written, the hint floats to the wrong place on a scrolled deck; E-R11-3 (`length >= 1`) still passes (it only counts existence), so the test would NOT catch the mispositioning. Add a position assertion to E-R11-3.

### G6 — `agentStampMap` re-runs `matchAnchor` per save; cost + a correctness subtlety — **MINOR**
`matchAnchor` (comments.js:181) can hit fallback levels 5 (relaxed tail-tag scan) and return a DIFFERENT element than originally anchored if content hashes collide. For stamping, a level-5 ambiguous match returns null (good — no stamp). But a level-4 same-parent match could stamp a sibling with identical text. Low probability in practice; flag that `agentStampMap` should stamp ONLY on the high-confidence anchor (hook or path+hash exact, levels 1–2), falling back to "context excerpt only / no stamp" for fuzzy matches, so a fuzzy re-resolution never stamps the wrong element. Tighten S3-13 wording. MINOR.
