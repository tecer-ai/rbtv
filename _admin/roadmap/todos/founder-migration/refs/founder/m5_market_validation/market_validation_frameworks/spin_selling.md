# SPIN Selling for Hypothesis Validation

**Purpose:** Execute systematic market validation conversations that test critical business assumptions from M2 using SPIN Selling methodology

**Prerequisites:** MUST complete leap_of_faith.md (M2) and product_overview.md (M1) before executing this framework.

---

## Framework Overview

Each SPIN question type maps to hypothesis testing:
- **Situation Questions** → Establish experimental baseline
- **Problem Questions** → Test Value Hypothesis (do customers have the pain we assume?)
- **Implication Questions** → Quantify risks identified in M2 analysis (how costly is the problem?)
- **Need-Payoff Questions** → Test Growth Hypothesis (will customers pay to solve this?)

When customers articulate their own problems and the value of solving them, you gain empirical evidence for or against your M2 assumptions. This is more rigorous than surveys or interviews—you're watching customers demonstrate willingness to engage with a solution.

---

## Task Structure

High-level view of framework execution:

| Task | Goal | Key Inputs | Key Outputs |
|------|------|------------|-------------|
| 1. Prepare Hypothesis-Based Questions | Design SPIN questions that test specific M2 assumptions | leap_of_faith.md, product_overview.md | Question set mapped to hypotheses |
| 2. Execute Situation Questions | Establish customer context baseline | Question set, customer research | Context facts, workflow understanding |
| 3. Conduct Problem Questions | Validate Value Hypothesis pain points | Situation baseline, M2 Value Hypothesis | Confirmed/refuted problems |
| 4. Develop Implication Questions | Quantify risks from M2 analysis | Problems, M2 risk matrix | Quantified costs, urgency data |
| 5. Guide Need-Payoff Questions | Test Growth Hypothesis viability | Implications, M2 Growth Hypothesis | Customer-stated value, buying signals |
| 6. Synthesize Validation Evidence | Connect findings to M2 assumptions | Conversation notes, M2 criteria | Validation results, pivot signals |

## Task 1: Prepare Hypothesis-Based Questions

**Goal:** Transform M2 leap of faith assumptions into testable SPIN questions

**Inputs:**

| Input | Source | Required |
|-------|--------|----------|
| leap_of_faith.md | M2 Validation milestone | Yes |
| product_overview.md | M1 Conception milestone | Yes |
| M2 risk assessment matrix | leap_of_faith.md | Yes |
| Customer research | Initial outreach, industry knowledge | No |

**Action:**

1. Read leap_of_faith.md—identify all Value Hypothesis and Growth Hypothesis assumptions
2. For each M2 assumption, write hypothesis statement: "If [assumption] is true, then customers will [observable behavior/statement]"
3. Map hypotheses to SPIN question types:
   - Value Hypothesis → Problem Questions
   - Growth Hypothesis → Need-Payoff Questions
   - Risk quantification → Implication Questions
4. Draft 5-7 Situation Questions (minimize—research first)
5. Draft 5-10 Problem Questions testing assumed pains
6. Draft 8-12 Implication Questions quantifying M2 risks
7. Draft 5-8 Need-Payoff Questions testing buying intent
8. Create question map: M2 Assumption → SPIN Question → Expected Answer

**Output:**

| Output | Format | Purpose |
|--------|--------|---------|
| Hypothesis-to-question map | Table | Links each M2 assumption to specific testable questions |
| SPIN question bank | Categorized list | Questions organized by type (S/P/I/N) |
| Expected validation signals | Checklist | What answers indicate hypothesis is true/false |

**Validation:**

- [ ] Every high-priority assumption from M2 maps to at least one SPIN question
- [ ] Each question has a clear "if true" and "if false" signal defined
- [ ] Problem Questions directly probe the pains described in Value Hypothesis
- [ ] Implication Questions quantify the risks flagged in M2 risk matrix
- [ ] Need-Payoff Questions test whether Growth Hypothesis assumptions hold

## Task 2: Execute Situation Questions

**Goal:** Establish customer context baseline without interrogating

**Inputs:**

| Input | Source | Required |
|-------|--------|----------|
| Question bank from Task 1 | Task 1 Output | Yes |
| Customer research | Pre-call research | Yes |
| Customer profile | CRM, LinkedIn, website | Yes |

**Action:**

1. Research beforehand to answer Situation Questions (company size, tools, industry)
2. Limit live Situation Questions to 2-4 that cannot be researched
3. Ask conversationally: "I understand you're using [tool]—how long have you been with that system?"
4. Listen for workflow details revealing bottlenecks
5. Note unexpected context changing question strategy
6. Transition quickly to Problem Questions

**Situation Question Templates:**

| # | Question Template | Adaptation Notes |
|---|-------------------|------------------|
| 1 | How many people are involved in [process from M2 context]? | Scale indicator for Growth Hypothesis |
| 2 | What tools do you currently use for [task related to product]? | Competitive baseline; switching cost assessment |
| 3 | Can you walk me through what happens when [trigger event]? | Workflow mapping; identifies pain point locations |
| 4 | How do you currently measure success in [relevant area]? | Reveals existing KPIs for validation |
| 5 | What volume of [items] do you handle per [timeframe]? | Quantification setup for Implication Questions |

**Output:**

| Output | Format | Purpose |
|--------|--------|---------|
| Customer context summary | Notes | Baseline facts about customer situation |
| Workflow map | Diagram or text | Visual of process flow showing handoffs |
| Unexplored areas | List | Topics to return to in Problem stage |

**Validation:**

- [ ] You can describe the customer's current process in their language
- [ ] You identified 2-3 specific areas where problems likely exist
- [ ] You spent less than 15% of conversation time on Situation Questions
- [ ] Customer feels heard, not interrogated

## Task 3: Conduct Problem Questions

**Goal:** Validate whether customers experience the Value Hypothesis pains

**Inputs:**

| Input | Source | Required |
|-------|--------|----------|
| Context from Task 2 | Task 2 Output | Yes |
| Value Hypothesis from M2 | leap_of_faith.md | Yes |
| Problem Questions from Task 1 | Task 1 Output | Yes |

**Action:**

1. For each M2 Value Hypothesis assumption, ask Problem Questions probing whether customers experience that pain
2. Use open-ended phrasing: "What challenges do you face with [process]?" not "Do you have trouble with [specific thing]?"
3. When customer mentions problem, ask follow-ups exploring scope: "Where else does that show up?" "How often?"
4. Note customer language—their words become messaging
5. Distinguish unprompted problems (real pain) from suggested problems (maybe pain)
6. If expected M2 problem not mentioned, probe once explicitly, then note "assumption not validated"
7. Spend 25-30% of conversation time in Problem exploration

**Problem Question Templates:**

| # | Question Template | Hypothesis Connection |
|---|-------------------|----------------------|
| 1 | What challenges do you face with [current process/tool]? | Opens exploration of Value Hypothesis pains |
| 2 | Where do bottlenecks occur in [workflow from Task 2]? | Tests assumption about process inefficiency |
| 3 | What happens when [process] fails or errors occur? | Validates assumption about failure costs |
| 4 | How much time does your team spend on [manual task]? | Quantifies labor assumption from M2 |
| 5 | What frustrates your team most about [current approach]? | Uncovers emotional pain; often reveals hidden problems |
| 6 | How consistent are results from [current process]? | Tests quality/reliability assumptions |
| 7 | What manual work do you wish you could eliminate? | Direct test of automation value assumption |

**Output:**

| Output | Format | Purpose |
|--------|--------|---------|
| Validated problems | List with evidence | Problems customer confirmed experiencing |
| Refuted assumptions | List | M2 assumptions customers did not confirm |
| New problems discovered | List | Pains not anticipated in M2 |
| Customer language log | Quotes | Exact phrases for messaging |

**Validation:**

- [ ] Customer mentioned at least 2 problems unprompted (not just agreeing with suggestions)
- [ ] You have specific examples of when problems occurred (not abstract)
- [ ] You noted which M2 Value Hypothesis assumptions are confirmed vs. refuted
- [ ] Customer's language and framing recorded for later use

## Task 4: Develop Implication Questions

**Goal:** Quantify risks identified in M2 analysis and build urgency

**Inputs:**

| Input | Source | Required |
|-------|--------|----------|
| Validated problems from Task 3 | Task 3 Output | Yes |
| M2 risk assessment matrix | leap_of_faith.md | Yes |
| Implication Questions from Task 1 | Task 1 Output | Yes |

**Action:**

1. For each validated problem, ask Implication Questions exploring consequences and ripple effects
2. Use M2 risk matrix—quantify "high impact" flagged risks
3. Chain implications: "What happens downstream when [problem] occurs?" then "How does that affect [goal]?"
4. Translate qualitative to quantitative: "Some time" → "40 hours/month" → "$X in labor" → "Delayed projects" → "Lost revenue"
5. Explore implications across dimensions:
   - Financial: Direct costs, opportunity costs, lost revenue
   - Strategic: Growth, competitive position, scalability
   - Operational: Capacity, error rates, bottlenecks
   - Customer: Satisfaction, churn, reputation
6. If customer says "it's not that bad," probe: "What if [problem] got worse over next year?"
7. Spend 35-40% of conversation time developing implications
8. Connect problems to customer's strategic goals

**Implication Question Templates:**

| # | Question Template | Risk Quantification Purpose |
|---|-------------------|----------------------------|
| 1 | How does [problem] affect your ability to [strategic goal]? | Links operational pain to strategic impact |
| 2 | What happens to [downstream process/customer] when [problem] occurs? | Reveals ripple effects that multiply severity |
| 3 | What's the cost when [problem] causes [delay/error/rework]? | Quantifies financial impact for ROI case |
| 4 | How does [bottleneck] impact capacity to handle growth? | Tests scalability assumption from Growth Hypothesis |
| 5 | What opportunities are you missing because resources are tied up in [manual work]? | Reveals opportunity cost beyond direct costs |
| 6 | How does [problem] affect customer experience or satisfaction? | Connects internal operations to customer impact |
| 7 | What would happen if [problem] got worse over next year? | Projects consequences to increase urgency |
| 8 | How much management time is spent dealing with issues from [problem]? | Captures hidden leadership cost |
| 9 | How does [problem] compare to what competitors deal with? | Introduces competitive pressure as implication |
| 10 | What's the impact on employee retention when [frustrating process] persists? | Links operational problems to talent/HR costs |

**Output:**

| Output | Format | Purpose |
|--------|--------|---------|
| Quantified costs | Table with estimates | Financial impact of each validated problem |
| Strategic impact assessment | Notes | How problems block strategic goals |
| Urgency indicators | List | Signals that problem is getting worse or intolerable |
| Competitive vulnerability | Analysis | Areas where competitors may exploit gaps |

**Validation:**

- [ ] At least 3 problems have quantified financial impact (even rough estimates)
- [ ] Customer connected problems to their strategic goals or growth plans
- [ ] You can articulate ROI case from customer's own data
- [ ] Customer expressed urgency or concern about consequences

## Task 5: Guide Need-Payoff Questions

**Goal:** Test Growth Hypothesis by having customers articulate value of solving problems

**Inputs:**

| Input | Source | Required |
|-------|--------|----------|
| Developed implications from Task 4 | Task 4 Output | Yes |
| M2 Growth Hypothesis | leap_of_faith.md | Yes |
| Need-Payoff Questions from Task 1 | Task 1 Output | Yes |

**Action:**

1. After problems and implications developed, ask Need-Payoff Questions prompting customer to state value of solving
2. Frame as "what if we could solve [problem]?" not "would you buy our product?"
3. Test Growth Hypothesis assumptions:
   - Will customers pay (how much)?
   - Will solution spread through organization or word-of-mouth?
   - Will early adopter interest transfer to mainstream buyers?
4. Listen for buying signals:
   - Customer asks "how does this work?" or "when could we start?"
   - Customer estimates value in concrete terms
   - Customer mentions budget or approval process
5. Note resistance or skepticism invalidating Growth Hypothesis
6. Spend 20-25% of conversation time on Need-Payoff
7. Record customer value articulation in exact words

**Need-Payoff Question Templates:**

| # | Question Template | Growth Hypothesis Test |
|---|-------------------|------------------------|
| 1 | If you could reduce [problem] by [X]%, what would that mean for your operation? | Tests whether customer values the improvement |
| 2 | How would your team's work change if [manual task] was automated? | Validates that solution creates meaningful change |
| 3 | What could your team accomplish if they weren't spending time on [repetitive work]? | Shifts from cost-saving to value-creation framing |
| 4 | How valuable would it be to have consistent, measurable results from [process]? | Tests assumption about analytics/measurement value |
| 5 | What would it be worth to solve [problem] permanently? | Invites customer to state monetary value |
| 6 | How would [solving problem] affect your ability to scale the business? | Tests scalability assumption from Growth Hypothesis |
| 7 | If you could see exactly where time and money were being lost in [process], how would that help? | Validates diagnostic value assumption |
| 8 | What impact would [improvement] have on your customers? | Tests whether customer-focused benefits resonate |

**Output:**

| Output | Format | Purpose |
|--------|--------|---------|
| Customer-stated value | Quotes | Exact language describing what solving would mean |
| Buying signals observed | List | Behaviors indicating purchase intent |
| Willingness-to-pay indicators | Notes | Signals about budget, urgency, priority |
| Resistance points | List | Objections or skepticism that may block sale |

**Validation:**

- [ ] Customer articulated value in their own words (not just agreeing with you)
- [ ] Customer provided concrete examples of what they could do if problem solved
- [ ] You noted signals about budget, timeline, or approval process
- [ ] Any resistance or objections documented for later analysis

## Task 6: Synthesize Validation Evidence

**Goal:** Connect SPIN conversation findings back to M2 assumptions and make persevere/pivot/kill decision

**Inputs:**

| Input | Source | Required |
|-------|--------|----------|
| All outputs from Tasks 2-5 | Task 2-5 Outputs | Yes |
| M2 leap_of_faith.md | M2 Validation | Yes |
| M2 kill criteria | leap_of_faith.md | Yes |

**Action:**

1. Create synthesis table:

   | M2 Assumption | SPIN Evidence | Validation Status | Confidence |
   |---------------|---------------|-------------------|------------|
   | [Assumption] | [Customer quotes, behaviors] | Validated/Refuted/Uncertain | High/Medium/Low |

2. For each Value Hypothesis assumption:
   - Did Problem Questions confirm customers have this pain?
   - Did Implication Questions reveal costs as high as M2 estimated?
   - Decision: Validated, Refuted, or Needs More Data

3. For each Growth Hypothesis assumption:
   - Did Need-Payoff Questions show customers value solving this?
   - Did customers signal willingness to pay?
   - Did conversation suggest viral/word-of-mouth potential?
   - Decision: Validated, Refuted, or Needs More Data

4. Compare findings to M2 kill criteria:
   - Kill criteria met or approached?
   - Do refuted assumptions invalidate business model?
   - Is pivot possible if core assumptions fail?

5. Count SPIN conversations vs. sample size needed
6. Make recommendation:
   - **Persevere:** Core assumptions validated; proceed to MVP
   - **Pivot:** Some assumptions refuted; viable alternative identified
   - **Kill:** Critical assumptions refuted; no viable business model
   - **Continue Testing:** Insufficient data; need more conversations

**Output:**

| Output | Format | Purpose |
|--------|--------|---------|
| Validation synthesis table | Table | Maps M2 assumptions to SPIN evidence |
| Persevere/Pivot/Kill recommendation | Document | Explicit decision with rationale |
| Unvalidated assumptions list | List | Assumptions that still need testing |
| Next actions | Checklist | What to do based on validation results |

**Validation:**

- [ ] Every M2 high-priority assumption has validation status (Validated/Refuted/Uncertain)
- [ ] You have specific customer quotes supporting validation decisions
- [ ] You compared findings to M2 kill criteria explicitly
- [ ] You made clear recommendation: Persevere, Pivot, Kill, or Continue Testing
- [ ] If Pivot recommended, you identified what would change

## Pitfalls

**Testing Without M2 Context**

Running SPIN without leap_of_faith.md analysis leads to unfocused discovery without validating critical assumptions.

**Instead:** MUST complete M2 leap of faith before M5 SPIN. Map each assumption to specific test questions.

---

**Accepting Vague or Hypothetical Answers**

"That would be nice" or "we'd probably use that" are NOT buying signals. Real validation requires concrete examples, quantified costs, demonstrated urgency.

**Example:** Customer says "we could save some time" → Push for: "How much time per week?" "What would your team do with that time?" "What's that worth?"

**Instead:** NEVER accept vague answers. If customer can't provide specifics, note "uncertain" not "validated."

---

**Skipping Implication Development**

Moving quickly from Problem to Need-Payoff misses critical urgency building. Without implications, problems remain theoretical.

**Instead:** MUST spend 35-40% of conversation time on Implication Questions. Build business case through customer articulation of costs.

---

**Asking Leading Questions**

"Don't you think it would be better if [your solution existed]?" gets false validation.

**Instead:** ALWAYS use open-ended Problem Questions: "What challenges do you face?" Note unprompted problems vs. suggested agreement.

---

**Conflating Early Adopters With Mainstream Customers**

First enthusiastic conversations may validate assumptions that fail with mainstream buyers.

**Instead:** MUST segment early adopter vs. mainstream conversations. Track validation separately. NEVER assume learnings transfer.

---

**Ignoring Refuted Assumptions**

Rationalizing refuted assumptions is confirmation bias.

**Instead:** ALWAYS treat refuted assumptions seriously. If multiple conversations refute critical assumption, M2 hypothesis is wrong. Update leap_of_faith.md and consider pivot or kill.

---

**Over-Presenting, Under-Listening**

Jumping to solution presentation interrupts hypothesis testing.

**Instead:** NEVER present solutions before Need-Payoff stage. Goal is validation, not closing. If customer asks mid-conversation: "I'd love to show you—let me understand your situation first."
