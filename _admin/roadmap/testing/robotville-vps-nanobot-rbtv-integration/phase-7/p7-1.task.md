# Task p7-1: Audit System Update Scripts for Project File Safety

## Goal

Audit `vps-sync-install.sh`, `vps-pull-rbtv.sh`, the post-merge Git hook, and the RBTV installer to verify that system updates (`git pull` + sync-install) never overwrite, delete, or corrupt project output files (`projects/`) or any user project data in the workspace.

## Execution Flow

### Phase: Understand

1. **MUST** read every file in the Context Files table above. Do not continue until all are read.
2. In the plan file, set this task's todo to `status: in_progress`.
3. Review shape.md for prior decisions about deploy automation (especially the p5-6 and VPS normalization entries).
4. Identify all paths the scripts touch during execution.

### Phase: Execute

1. **Trace** every file operation in `vps-sync-install.sh`:
   - What directories does it create, copy to, or delete from?
   - Does the mirror sync step touch `projects/`?
   - Does the sparse-checkout cleanup remove any project data?
   - Does `rsync --delete` for skills remove anything outside `skills/`?
2. **Trace** `vps-pull-rbtv.sh` for any destructive operations.
3. **Trace** the post-merge hook for any side effects beyond calling sync-install.
4. **Trace** the RBTV installer (`_config/install-rbtv.py`) for any operations on `projects/`.
5. **Document** findings: which paths are safe, which paths are at risk.
6. **If any risk found:** Fix the script to exclude project data paths. Add explicit safeguards (e.g., `--exclude 'projects'` in rsync, existence checks before destructive operations).

**Discovery Handling:**
- If simple discovery (<5 min to resolve): Resolve immediately, document in shape.md
- If complex discovery: Add new task to plan, notify user of task addition

### Phase: Validate

1. Verify no script operation touches `projects/` or workspace project data.
2. Verify any fixes applied are idempotent and don't break the update flow.
3. Verify the deploy runbook accurately reflects the current script behavior.

### Phase: Close
