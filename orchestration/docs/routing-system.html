Developer reference

How the router picks which model gets a task.

A deterministic, stdlib-only selector. Give it a task profile; it returns one (model, variant, carrier) pick — by gating on capability floors, ranking cheapest-capable first, then raising the floor for pinned roles. Same script at plan time and run time, so the two can never disagree.

GATE → RANK → PIN integer 1–7 axes election-driven stdlib only

The full flow

Eight ordered stages

The three headline stages are GATE → RANK → PIN. The rest prepare the candidate set and finish the pick. route() returns one of route, self_execute, or halt_seam.

Stage 4 · GATE

Boundedness, floors, and role-eligibility

Boundedness — how fully the task is spelled out — is the first thing the router reads. It scopes the candidate set and sets the reasoning floor and effort.

BandScopeReasoning floorCoding floorEffort
fully-bounded
every edge written in		all eligible variants11low
partially-bounded
shape clear, some calls open		Claude mid-tier and up (reasoning ≥ 6)64medium
unbounded

Floors are byte-for-byte integer compares against the 1–7 axes — no value-lookup table. Constants: REASONING_FLOOR_BY_BAND, CODING_FLOOR_BY_BAND.

reasoning / coding

1 = weakest, 7 = strongest. Used as score ≥ floor at GATE and as the capability tiebreak at RANK.

cost

1 = cheapest, 7 = priciest. The primary RANK key, ascending — so cheapest-capable wins. Bin 7 ranks last and is never auto-picked on a tie.

why integers, not 0–100

Re-derived from one external methodology so they stay comparable across models and don't rot. A coarse floor-gate, not a fine ladder.

Role-eligibility is independent of the coding score

routable_for is an allow-list over a closed role vocabulary: bounded-code unbounded-code reasoning web-research text-synthesis other.

the rule

Absent/empty → eligible for every leaf (back-compat). Present → eligible only for the listed roles. Code leaves are gated on routable_for membership of a code role — never on the coding integer.

the consequence

An API chat worker can carry an honest coding score of 3–4 yet never receive a code leaf, because its routable_for omits the code roles. An honest score never re-enables an ineligible route. Unknown role string → not-matching, never a crash.

⊕Routing via the other catch-all records the task's instructions (_build_other_routing_audit()) so under-served task types surface for promotion to a first-class role.
Stage 6 · RANK

Cheapest-capable, deterministically

_rank() imposes a total order over the survivors. Cost is the primary key, ascending — so two equal-cost code survivors are ordered directly by their coding integer; there is no separate sub-rank step.

Stage 7 · PIN

Floors that raise, never lower

Pins apply after ranking. Defined in _PINNED_FLOORS, applied by _apply_pinned_role_floor().

Pinned roleFloor
reviewer≥ executor reasoning + 1, floor Sonnet (reasoning 6), never Haiku. reviews_external_cli_code: true forces Opus.
debugany code-eligible executor with reasoning ≥ 7.
commitAgent-tool Claude Sonnet; if no Claude is available, the strongest-reasoner across all elected packages (excluding Haiku), cost ignored.
stakes tier-up

A profile with stakes_tier: tier_up raises the boundedness band one level and re-resolves (re-scope, re-filter, re-rank) over the full enumeration.

empty-pipeline fallback

When the band-scoped pipeline yields zero candidates but a pinned_role is set, the pin's floor computation runs over the full pre-scope roster before returning an error — so e.g. a reviewer can still be filled.

Change the profile and the same eight stages run: a fully-bounded mechanical job needs only reasoning 1, so a cheap executor survives and wins on cost; an unbounded design task demands reasoning 7, so only the top tier passes. The manifests never change — the requirements do.

On disk

Package layout & election

orchestration/models/ {model}/ manifest.yaml # capability + routing inputs (the only routing source) delta.md # model-specific render source manual.md # GENERATED dispatch manual — never hand-edit manifest-schema.md / manifest-template.yaml # the schema + skeleton model-plans-schema.md / model-plans-example.yaml route.py / test_route.py / render-manuals.py _api/ _fixture/ mirror/ # infra dirs — never enumerated
election is authoritative

route.py enumerates only the packages in rbtv.json → model_packages, and (for a multi-backend package) only the backends in model_variants. Anything absent is skipped at enumeration. A --models-dir override disables the election filter.

the availability block is recall-only

The <!-- ORCH:AVAILABILITY --> block in skills/orchestrating/core-protocol.md is installer-written (last-install-wins) — it tells the skill what is routable. It is not a routing input; route.py reads election from rbtv.json.

Generated manuals

The render pipeline

A model's dispatch manual is composed — never hand-written — from the shared wrapper template plus the model's delta.md, via an HTML-comment marker grammar (RENDER:BEGIN/END, RENDER:INSERT, RENDER:DELTA). Rendering is deterministic (write-if-changed, no timestamps).

# from repo root python orchestration/models/render-manuals.py # regenerate all manuals python orchestration/models/render-manuals.py --check # report drift, write nothing → exit 0 when every manual is fresh · exit 1 when any is stale
✗Never edit manual.md directly — it is overwritten on the next render. To change cross-model behavior edit the wrapper template (skills/orchestrating/cards/dispatch-wrapper.md); to change model-specific behavior edit that model's delta.md; then re-render. --check must exit 0.
Verification

Running the tests

# canonical — from the repo root python -m pytest orchestration/models/test_route.py
!Gotcha — python orchestration/models/test_route.py collects ZERO tests and exits 0 with no output. The tests take pytest fixtures (tmp_path) and the file has no if __name__ == "__main__" block, so running it as a plain script just loads the test classes without invoking anything. Always run it through pytest. This silent no-op is the gap this document exists to close.

The suite covers: the reference profiles (fully / partially / unbounded), the boundedness keystone, determinism, API-key availability, plan caps (parse · apply · graceful skip), halt seams, self-execute, the --explain trace, failure modes, stakes & pinned-role floors, the Haiku guard, and a live-corpus enumeration.

Per-deployment limits

Plan-size context caps

A deployment can cap a model's usable context below its true ceiling (e.g. a subscription tier). The cap file is cap-only — context_window per model, nothing else; cost stays board-derived in the manifests.

the pointer

rbtv.json → model_plans_file points at the cap file. Shape & fields: model-plans-schema.md; example: model-plans-example.yaml.

the application

_apply_plan_caps() sets each variant's effective window to min(manifest window, plan cap) — downward only.

never required

If the pointer is absent or the file is unreadable/unparseable, routing proceeds on the manifest windows unchanged.

Procedure

Add or migrate a model

Elect the package — add it to model_packages in rbtv.json (installer picker or --model-packages). For a multi-backend package, set its model_variants subset.
Author models/{model}/manifest.yaml from manifest-template.yaml, filling every required field with an evidence-traceable value per manifest-schema.md. Set the 1–7 axes from the same external methodology as the other models — keep them comparable.
Write delta.md with the model's dispatch specifics, including the required invocation section.
Render the manual: python orchestration/models/render-manuals.py, then confirm --check exits 0.
Add/extend tests in test_route.py for any new routing behavior.
Run the suite: python -m pytest orchestration/models/test_route.py — all green.
Keep docs in sync (repo hard rule): update modules/orchestration.md, the matrix reference if a routable variant changed, and admin/install/module-manifest.json if a package was added/removed/renamed.
