# Problem-Solution Fit Framework

**Purpose:** Validate that your proposed solution is a natural, credible response to a well-understood problem in a real customer context.

---

## Framework Overview

Problem-Solution Fit Canvas tests whether the solution truly fits a specific customer problem in a concrete context. The framework forces:

1. **Narrow scope** — One primary segment and one recurring situation per canvas
2. **Problem grounding** — Map problem as experienced, not as abstract category
3. **Behaviour awareness** — Document current behaviours and alternatives
4. **Constraint respect** — Surface limitations that reshape solution design
5. **Solution traceability** — Every solution element traces to behaviour or constraint

---

## Canvas Blocks

| Block | Purpose | Source |
|-------|---------|--------|
| Problem-Space Brief | Locks in who and what context | Working Backwards + JTBD |
| Problem | Recurring pain described in situ | Discovery questioning |
| Triggers | When and why customers act | JTBD struggling moments |
| Emotions (Before/After) | Functional and emotional jobs | PR narrative + JTBD |
| Available Solutions | Existing alternatives and drawbacks | Market observation |
| Behaviour | Coping strategies and workarounds | Customer research |
| Channels of Behaviour | Where problem-related actions occur | Workflow mapping |
| Customer Constraints | Limitations affecting adoption | Budget, skills, politics |
| Our Solution | Solution concept (not features) | PR/FAQ solution |
| Critical Assumptions | Tagged beliefs requiring validation | Internal FAQ + analysis |

---

## Key Principles

### Scope Enforcement

NEVER cover multiple customer types in one canvas. Enforce narrow scope — one primary segment and one recurring situation per canvas. If multiple segments emerge, propose splitting into separate canvases.

### Solution Definition

NEVER jump from problem to feature backlog. Force each solution element to trace back to:
- A behaviour it replaces or augments
- An alternative it improves upon
- A constraint it respects or relaxes

### Constraint Surfacing

NEVER downplay constraints. Types to surface:
- Financial (budget, cost structure)
- Time and attention (competing priorities)
- Skills and tools (technical literacy)
- Organisational (approvals, compliance, IT/security)

## Assumption Categories

| Category | Examples |
|----------|----------|
| Behavioural | Customer experiences problem with certain frequency; willing to change behaviour |
| Technical | Feasibility and performance requirements |
| Economic | Pricing, costs, market size, buying process |

---

## Validation Approach

For each assumption:
1. Write "This assumes that..." statement
2. Classify as behavioural, technical, or economic
3. Prioritise by impact if wrong
4. Identify concrete tests or signals that would falsify

Top 3-5 assumptions should feel uncomfortable — if wrong, the canvas collapses.
