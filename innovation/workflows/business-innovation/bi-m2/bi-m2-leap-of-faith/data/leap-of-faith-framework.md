# Leap of Faith Framework

**Purpose:** Transform the diffuse cloud of assumptions from M1 Conception into an explicit, ranked inventory that drives the entire M2 validation agenda.

---

## Core Concept

Every new product rests on untested beliefs. Most founders treat assumptions as background noise rather than explicit risks. Leap of Faith forces you to:

1. **Surface** every assumption hiding in M1 artefacts (explicit and implicit)
2. **Classify** each as Value Hypothesis or Growth Hypothesis
3. **Rank** by Impact × Uncertainty to identify true leap-of-faith assumptions
4. **Define** observable signals that validate or invalidate each
5. **Pre-commit** to kill/pivot/persevere decisions before sunk-cost bias sets in

The output is not a plan to be right. It is a plan to **learn fast and fail cheaply**.

---

## Hypothesis Types

### Value Hypothesis

Assumptions that, if true, mean customers experience the stated pain and will hire your solution to make progress:

| Sub-Category | Question Answered |
|--------------|-------------------|
| Problem Existence | Does this pain actually exist? |
| Problem Severity | Is it painful enough to motivate action? |
| Solution Fit | Does our solution address the pain? |
| Willingness to Switch | Will they switch from current workaround? |
| Retention & Habit | Will they keep using it? |

### Growth Hypothesis

Assumptions that, if true, mean the business can acquire and retain customers profitably:

| Sub-Category | Question Answered |
|--------------|-------------------|
| Channel Effectiveness | Can we reach customers through this channel? |
| Acquisition Cost | Can we afford to acquire customers this way? |
| Conversion Rate | Will prospects convert at this rate? |
| Pricing & WTP | Will they pay this price? |
| Revenue Model | Will this revenue structure work? |
| Retention Economics | Will renewals/retention sustain the business? |
| Referral & Virality | Will customers refer others? |
| Market Size | Is the market large enough? |

## Scoring Framework

### Impact Score (1-5)

| Score | Meaning |
|-------|---------|
| 5 | If wrong, no viable business exists |
| 4 | If wrong, major pivot required |
| 3 | If wrong, significant redesign needed |
| 2 | If wrong, moderate adjustment required |
| 1 | If wrong, minor parameter change |

### Uncertainty Score (1-5)

| Score | Meaning |
|-------|---------|
| 5 | Pure speculation, founder intuition only |
| 4 | Indirect or analogical evidence only |
| 3 | Some evidence but not from target segment |
| 2 | Direct evidence but limited sample |
| 1 | Well-established with multiple data points |

**Priority Score** = Impact × Uncertainty

Top-right quadrant (High Impact, High Uncertainty) = True leap-of-faith assumptions

## Kill Criteria Design

Pre-commit to rational responses before evidence arrives:

| Outcome | Trigger |
|---------|---------|
| **Kill** | What combination of invalidated assumptions would stop the venture? |
| **Pivot** | What evidence distinguishes "wrong solution" from "wrong problem"? |
| **Persevere** | What positive evidence (not just absence of negative) is required? |

**Aggregate criteria:** How many Value assumptions can fail? How many Growth assumptions?

---

## Validation Signal Spec

For each top assumption, define:

| Field | Description |
|-------|-------------|
| Validation Signal | Observable outcome that increases confidence |
| Invalidation Signal | Observable outcome that decreases confidence |
| Signal Threshold | Minimum quantity/quality of evidence |
| Collection Method | Interview, survey, smoke test, research, etc. |
| Timeline | By when must this signal be collected? |
| Owner | Who is responsible? |

| Framework | Assumption Types It Validates |
|-----------|------------------------------|
| Assumption Mapping (M2) | All — plots the matrix visually |
| TAM/SAM/SOM (M2) | Market Size, Channel Effectiveness |
| Unit Economics (M2) | Pricing, LTV, CAC, Retention Economics |
| Pre-mortem (M2) | Top assumptions become failure modes |
| SPIN Selling (M5) | Problem Existence, Severity, Willingness to Switch |
| Smoke Test (M5) | Demand, Channel, Conversion |

## Pitfalls

1. **Harvesting only explicit assumptions** — The most dangerous ones are implicit
2. **All Value or all Growth** — Must have assumptions in both categories
3. **Scoring to avoid discomfort** — Challenge every low uncertainty score
4. **Unfalsifiable kill criteria** — Must be achievable within validation timeline
5. **Static backlog** — Update after every M2/M5 activity
